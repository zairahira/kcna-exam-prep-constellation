## Observability vs monitoring

**Monitoring** tells you _when_ something is wrong. You define thresholds and
alerts in advance (CPU > 90%, error rate > 5%), and the system notifies you
when they are breached.

**Observability** lets you ask _why_ something is wrong - even for problems
you did not anticipate. An observable system exposes enough internal state
that you can investigate novel failures without deploying new instrumentation.

Monitoring is a subset of observability. You need monitoring, but in complex
distributed systems, monitoring alone is not enough.


## What is telemetry?

Telemetry is the automated collection and transmission of data from your
applications and infrastructure. It includes the three pillars - logs,
metrics, and traces - plus any other signals your system emits.

In cloud native environments with hundreds of microservices running across
many nodes, telemetry is the only way to understand what is happening inside
the system.


## The three pillars

### 1. Logs - what happened

A **log** is an immutable, timestamped record of a discrete event.

- Container logs go to `stdout`/`stderr` and are collected by Kubernetes
- Logs answer questions like "What error did the payment service return at
  14:32?"
- Best when **structured** (JSON format) so they can be parsed and queried
- High volume - needs aggregation and indexing (more on this in the logging
  lesson)

### 2. Metrics - how much / how often

A **metric** is a numeric measurement recorded over time - a time series.

- Examples: request count, error rate, CPU usage, memory consumption, latency
  percentiles
- Metrics answer questions like "How many requests per second is the API
  handling?" or "What is the p99 latency?"
- Low storage cost (just numbers + timestamps), good for dashboards and
  alerting
- Prometheus is the dominant metrics tool in cloud native (covered next lesson)

### 3. Traces - where time was spent

A **trace** follows a single request as it travels across multiple services.

- A trace is made up of **spans** - each span represents work done by one
  service
- Traces answer questions like "Why did this checkout request take 4 seconds?
  Which service was slow?"
- Essential for debugging latency in microservice architectures
- Tools: Jaeger, Zipkin, OpenTelemetry (covered in the tracing lesson)


## How the pillars complement each other

No single pillar is enough on its own:

| Signal | Strength | Weakness |
|--------|----------|----------|
| Logs | Rich context, exact error messages | High volume, hard to correlate across services |
| Metrics | Cheap, great for alerting and trends | No context - tells you _what_ but not _why_ |
| Traces | Shows the full request path | Sampling means you may not capture every request |

A typical debugging flow uses all three:

1. **Metrics** alert you that error rate spiked
2. **Traces** show that the slow requests all pass through the inventory service
3. **Logs** from the inventory service reveal a database connection timeout


## Exam-relevant summary

- Observability = ability to understand internal state from external outputs
- Three pillars: **logs** (events), **metrics** (numbers over time), **traces** (request paths)
- Telemetry = collecting and transmitting all of this data automatically
- In distributed systems, you need all three pillars working together
- The CNCF ecosystem has graduated/incubating projects for each pillar
  (Prometheus for metrics, Fluentd for logs, Jaeger for traces, OpenTelemetry
  for instrumentation)
