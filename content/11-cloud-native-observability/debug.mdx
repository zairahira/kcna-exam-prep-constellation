<DebugPuzzle id="m11-d2" syllabusIds={["OBS-LOGS"]}
  prompt={<p>A Pod writes application logs to <code>/var/log/myapp.log</code> inside the container. kubectl logs shows nothing. Why?</p>}
  artifact={<TerminalBlock command="kubectl logs analytics-api-7b9d8">{"\n"}</TerminalBlock>}
  choices={[
    { id: "a", label: "kubectl logs reads only what the container wrote to stdout/stderr. Logs to a file inside the container aren't captured. The Twelve-Factor way is to log to stdout; if the app really must log to a file, use a sidecar to tail it to stdout." },
    { id: "b", label: "Logs take 10 minutes to propagate." },
    { id: "c", label: "You need --from-file=/var/log/myapp.log." },
    { id: "d", label: "kubectl logs requires RBAC on Secrets." },
  ]}
  correctId="a"
  explanation={<p>Container logs in Kubernetes = whatever the container wrote to stdout/stderr. Internal log files are invisible to <code>kubectl logs</code> and to node-level log collectors (Fluentd/Fluent Bit). Fix: configure the app to write to stdout, or run a sidecar that tails the file to stdout.</p>}
/>

<DebugPuzzle id="m11-d3" syllabusIds={["OBS-TRACE"]}
  prompt={<p>A request goes through 5 microservices. Traces for each individual service show up in Jaeger, but they're all separate traces - not stitched together. What's the most likely cause?</p>}
  artifact={null}
  choices={[
    { id: "a", label: "Context propagation isn't happening. Each service is starting its own trace instead of continuing the parent trace via the traceparent header (or equivalent). Check the HTTP client instrumentation on each service." },
    { id: "b", label: "Jaeger only supports single-service traces." },
    { id: "c", label: "You need to merge traces manually in the UI." },
    { id: "d", label: "The OpenTelemetry collector is down." },
  ]}
  correctId="a"
  explanation={<p>For distributed traces, every service must propagate the trace context to downstream calls - usually via the W3C <code>traceparent</code> header. If a service starts a new trace instead of continuing the parent, you get disconnected traces. OpenTelemetry&apos;s auto-instrumentation handles this for common HTTP clients, but custom transports or a gap in instrumentation can break the chain.</p>}
