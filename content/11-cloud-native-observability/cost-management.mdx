## Why cost is an observability concern

Cloud native platforms make it easy to spin up infrastructure.
They&apos;re equally good at wasting it. Unattended dev clusters,
oversized production workloads, forgotten LoadBalancers, and orphaned
PVs add up fast.

Cost management in cloud native is about **visibility** and
**attribution** - knowing what you&apos;re spending and which team,
service, or feature is responsible.


## Common sources of waste

### Oversized requests

A common pattern: developers set `resources.requests.cpu: 2000m` "to be
safe." The scheduler then reserves 2 CPUs of node capacity even though
the app uses 200m. Multiply across replicas and services - you&apos;re
paying for capacity that&apos;s never used.

Fix: right-size requests based on observed usage (use metrics). Tools
like **VPA in recommendation mode** can suggest values.

### Idle workloads

Dev and staging environments running 24/7. Preview environments not
cleaned up after PR merge. Jobs left running.

Fix: cluster autoscaler, scale-to-zero where possible, automated
environment teardown policies.

### Orphaned resources

PVs retained after PVC deletion, LoadBalancer Services that someone
forgot to delete (they cost $$ per month while they exist), unused
Ingress controllers.

Fix: regular audits, cost attribution by label/namespace to catch
oddities.

### Storage class choices

Using `fast-ssd` for everything, including workloads that would be fine
on `standard`. Storage pricing varies widely.

Fix: understand your StorageClasses and pick deliberately.


## Cost observability tools

### Kubecost

Commercial product with an open-source core. Analyses Kubernetes
resource usage and maps it to cloud billing data to show cost per
namespace, Deployment, label, etc.

### OpenCost

**OpenCost** is the open-source specification and reference
implementation for Kubernetes cost monitoring (CNCF Sandbox).
Kubecost builds on OpenCost. Provides the standard way to compute
cluster cost at namespace/workload granularity.

### Cloud provider cost tools

AWS Cost Explorer, GCP Billing Reports, Azure Cost Management give you
cloud-account-level spend but not per-Kubernetes-workload detail.
Combine with Kubecost/OpenCost for attribution.


## FinOps

**FinOps** is the operating model for cloud financial management. Key
principles:

- **Collaboration** between engineering, finance, and product on cost.
- **Decisions are value-based** - cost is a trade-off against feature
  velocity, performance, reliability.
- **Visibility drives accountability** - teams that can see their spend
  make better choices.
- **Continuous improvement** - FinOps is a practice, not a one-time
  project.

The **FinOps Foundation** is part of the Linux Foundation (sibling to
CNCF).


## Practical starting points

1. **Label everything** - team, environment, service, cost-center. Your
   cost tools are only as good as your labels.
2. **Turn on cost dashboards** - Kubecost / OpenCost at minimum.
3. **Right-size requests and limits** - use VPA recommendations or
   analyse metrics manually.
4. **Set budgets and alerts** - at the cluster or namespace level.
5. **Clean up regularly** - automate deletion of preview envs, scan for
   orphaned LBs and PVs.
