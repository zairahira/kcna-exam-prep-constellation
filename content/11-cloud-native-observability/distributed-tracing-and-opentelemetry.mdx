## What tracing solves

Metrics tell you **how much** and **how often**. Logs tell you **what
happened**. Neither answers the question "in this specific slow request,
which downstream service ate the time?"

**Distributed tracing** captures the full path of a request across
services - every hop, every call, every DB query, with timings and
metadata. For microservice systems, it&apos;s indispensable.


## Core concepts

### Trace

A **trace** represents one end-to-end request. It has a unique trace ID.

### Span

A **span** is one unit of work within a trace - e.g. one service
handling one call. Spans have:

- A start and end time
- A name (the operation)
- A parent span (unless it&apos;s the root)
- Attributes (key-value metadata)
- Events and links

A trace is a tree (sometimes a DAG) of spans.

### Context propagation

When service A calls service B, A must pass the trace ID and parent
span ID along, so B can create a child span on the same trace. This is
typically done via HTTP headers (`traceparent`, `tracestate` per the W3C
Trace Context standard).

Without context propagation, you get disconnected traces - effectively
just local logs.


## OpenTelemetry

**OpenTelemetry** (OTel - CNCF Incubating) is the vendor-neutral standard
for instrumenting applications and collecting telemetry data - metrics,
logs, and traces.

It was created by merging two earlier projects:

- **OpenTracing** - tracing API spec
- **OpenCensus** - telemetry libraries from Google

Today OTel provides:

- **APIs and SDKs** for most popular languages (Go, Java, Python,
  Node.js, .NET, etc.)
- **Instrumentation libraries** that automatically trace HTTP clients,
  DB drivers, gRPC, etc.
- The **OpenTelemetry Collector** - a vendor-neutral agent that receives,
  processes, and exports telemetry to any backend.

The collector&apos;s job: take data in (via OTLP, Jaeger, Prometheus,
Fluentd formats, etc.), process it (filter, batch, sample), and ship to
any backend.


## Tracing backends

OpenTelemetry specifies the **data**, not the backend. Backends that
receive and display traces:

- **Jaeger** - CNCF Graduated. Originally from Uber. Open source, wide
  adoption.
- **Zipkin** - the original open-source tracer from Twitter.
- **Tempo** - Grafana&apos;s traces store.
- Commercial: Honeycomb, Datadog, New Relic, Lightstep, etc.


## When tracing matters

**Worth the effort:**

- Microservice systems (many services per request)
- Performance investigations (where does time go?)
- Understanding production dependencies
- Debugging rare tail-latency issues

**Probably not needed:**

- Small monoliths (logs + metrics cover you)
- Early-stage apps with few services
- Batch pipelines with simple structure


## Sampling

Tracing every request in a high-traffic system is expensive. **Head
sampling** decides at the trace root whether to keep. **Tail sampling**
decides after seeing the whole trace (useful to keep all errors /
slow requests). OpenTelemetry supports both.
