## What Prometheus is

**Prometheus** is a time-series database and monitoring system. It's the
de facto standard for metrics in cloud native, and was the **second**
project to graduate from the CNCF (after Kubernetes itself).

Key characteristics:

- **Pull-based** - Prometheus scrapes metrics from HTTP endpoints your
  services expose. No agent pushes to Prometheus.
- **Time-series data model** - every sample has a metric name, a set of
  key/value labels, a timestamp, and a numeric value.
- **PromQL** - a query language for slicing and aggregating metrics.
- **Federation** - Prometheus servers can scrape each other for
  hierarchical setups.


## The pull model

Your service exposes an HTTP endpoint (by convention `/metrics`) that
returns metrics in plain-text Prometheus format. A scrape config tells
Prometheus which endpoints to hit and how often.

<TerminalBlock label="/metrics endpoint (example)">
{`# HELP http_requests_total Total HTTP requests
# TYPE http_requests_total counter
http_requests_total{method="GET",status="200"} 12845
http_requests_total{method="POST",status="201"} 341

# HELP process_memory_bytes Process memory in use
# TYPE process_memory_bytes gauge
process_memory_bytes 1.2e8`}
</TerminalBlock>

Pull model benefits: Prometheus knows which targets are up (failure to
scrape = the target is down). No credentials to push; scrape with
service-discovery.


## Metric types

| Type | Use for |
|------|---------|
| **Counter** | Monotonically increasing values (requests served, errors, bytes sent). Never decreases; resets on process restart. |
| **Gauge** | Values that go up and down (memory in use, queue depth, active connections). |
| **Histogram** | Distributions. Records counts in predefined buckets. Good for latency. |
| **Summary** | Like histogram, but computes quantiles in the client. Less flexible than histogram for aggregation. |


## PromQL basics

<TerminalBlock>
{`# Current value\nprocess_memory_bytes\n\n# Rate over 5m (requests per second)\nrate(http_requests_total[5m])\n\n# Filter by label\nrate(http_requests_total{status="500"}[5m])\n\n# Aggregate across Pods\nsum by (service) (rate(http_requests_total[5m]))`}
</TerminalBlock>

PromQL is worth investing in. Most alerting and dashboarding in cloud
native runs on it.


## Exporters

Most software doesn&apos;t natively expose Prometheus metrics. An
**exporter** is a small daemon that reads some source (OS stats, MySQL
internals, Redis, etc.) and translates them to Prometheus format.

Common exporters:

- **node_exporter** - host-level metrics (CPU, memory, disk, network) for
  every node.
- **kube-state-metrics** - Kubernetes object state (counts of Pods in
  each phase, desired vs actual replicas, etc.). Distinct from
  metrics-server (which provides resource metrics for HPA).
- **Database exporters** - mysqld_exporter, postgres_exporter,
  redis_exporter.


## The rest of the stack

Prometheus alone gives you metrics + queries. Two companions:

- **Alertmanager** - receives alerts from Prometheus, deduplicates,
  groups, routes to destinations (PagerDuty, Slack, email).
- **Grafana** - visualisation. Not a CNCF project but universally paired
  with Prometheus for dashboards.

A typical stack:

```
App → /metrics → Prometheus → Grafana (dashboards)
                       ↓
                  Alertmanager → PagerDuty / Slack
```


## Long-term storage

Prometheus stores data locally with limited retention by design. For
long-term storage and multi-cluster federation, projects like **Thanos**
(CNCF Incubating) and **Cortex** (CNCF Incubating - now part of
**Grafana Mimir**) sit on top of Prometheus.
