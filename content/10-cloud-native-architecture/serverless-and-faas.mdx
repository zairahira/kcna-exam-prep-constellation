## What serverless means

"Serverless" is a misnomer - servers still exist. What changes is the
**operator model**: you don&apos;t provision, patch, or scale them. The
platform does.

Key characteristics:

- **No server management** - you don&apos;t think in terms of VMs or even
  containers; you hand over code (or a container) and it runs.
- **Scale-to-zero** - idle workloads consume no capacity. The platform
  spins them up when requests arrive.
- **Event-driven** - serverless functions typically run in response to
  events (HTTP request, queue message, file upload).
- **Fine-grained billing** - you pay per invocation and duration, not for
  idle VMs.


## FaaS vs. serverless containers

Two flavours of serverless at the platform level:

### Functions-as-a-Service (FaaS)

You deploy **functions** - small bits of code with a defined entry point.
The platform packages them and handles scaling. Examples:

- **AWS Lambda**
- **Google Cloud Functions**
- **Azure Functions**

FaaS is great for simple event-driven logic. Constraints: cold starts,
execution time limits, limited runtime environment.

### Serverless containers

You hand over a **container image** and configure scale-to-zero plus
event-driven invocation. More flexible than FaaS - any language, any
dependencies. Examples:

- **Google Cloud Run** - managed serverless containers.
- **AWS Fargate** (semi-serverless container compute).
- **Azure Container Apps**.


## Cloud native serverless projects

### Knative

**Knative** (CNCF Incubating) brings serverless semantics to Kubernetes.
Two components:

- **Knative Serving** - deploy a container, get auto-scaling (including
  scale-to-zero), traffic splitting, revisions.
- **Knative Eventing** - event sources, brokers, triggers that route
  events to workloads.

Knative powers Google Cloud Run, IBM Cloud Code Engine, and other managed
serverless-on-Kubernetes offerings.

### OpenFaaS

**OpenFaaS** is a FaaS platform that runs on Kubernetes. Simpler than
Knative; opinionated around "functions as containers."


## When serverless fits

**Good fit:**
- Event-driven processing (webhooks, queue consumers, image resizers)
- Sporadic or spiky workloads where idle cost matters
- Simple APIs with uneven traffic

**Not a good fit:**
- Long-running, steady workloads (a ~24/7-busy service costs more
  serverless than always-on)
- Latency-sensitive workloads where cold starts are unacceptable
- Workloads with complex local state

Serverless is a tool in the cloud native toolbox - not the universal
answer. Many production systems use a mix: always-on Deployments for
hot paths, serverless for event processing.
