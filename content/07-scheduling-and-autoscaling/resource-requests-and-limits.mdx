## Why resource management matters

Without resource configuration, a single Pod could consume all the CPU and
memory on a node, starving other workloads. Kubernetes uses **requests** and
**limits** to prevent this.


## Requests

A **request** is the amount of CPU or memory that Kubernetes **guarantees**
to a container. The scheduler uses requests to decide which node has enough
room for the Pod.

<TerminalBlock label="container with requests">
{`containers:
- name: app
  image: myapp:1.0
  resources:
    requests:
      cpu: "250m"
      memory: "128Mi"`}
</TerminalBlock>

- `250m` = 250 millicores = 0.25 CPU cores
- `128Mi` = 128 mebibytes of memory

The scheduler looks at each node's **allocatable** resources minus what is
already requested by other Pods. If a node has 2 CPUs and 1900m is already
requested, a Pod requesting 250m won't fit - the Pod stays `Pending`.


## Limits

A **limit** is the maximum amount of CPU or memory a container is allowed to
use. The kubelet enforces limits at runtime.

<TerminalBlock label="container with requests and limits">
{`containers:
- name: app
  image: myapp:1.0
  resources:
    requests:
      cpu: "250m"
      memory: "128Mi"
    limits:
      cpu: "500m"
      memory: "256Mi"`}
</TerminalBlock>

### What happens when limits are exceeded?

| Resource | Behavior when exceeded |
|----------|----------------------|
| **CPU** | The container is **throttled** - it gets less CPU time. It is not killed. |
| **Memory** | The container is **OOMKilled** (Out Of Memory Killed) by the kernel. Kubernetes restarts it according to the Pod's `restartPolicy`. |

This is a key distinction: CPU is **compressible** (you can give less),
memory is **incompressible** (you either have it or you don't).


## Requests vs limits

| | Requests | Limits |
|---|---|---|
| Purpose | Scheduling guarantee | Runtime ceiling |
| Who uses it | kube-scheduler | kubelet / kernel |
| Effect if not met | Pod stays `Pending` | CPU throttled or OOMKilled |
| Required? | No, but strongly recommended | No |

If you set limits but not requests, Kubernetes automatically sets requests
equal to limits. If you set neither, the container has no guarantees and no
ceiling.


## CPU and memory units

### CPU

| Notation | Meaning |
|----------|---------|
| `1` | 1 full CPU core |
| `500m` | 500 millicores = 0.5 CPU |
| `100m` | 100 millicores = 0.1 CPU |

### Memory

| Notation | Meaning |
|----------|---------|
| `128Mi` | 128 mebibytes (binary, 1 Mi = 1024 Ki) |
| `1Gi` | 1 gibibyte |
| `128M` | 128 megabytes (decimal, 1 M = 1000 K) |

Use `Mi` and `Gi` (binary) - they are more common in Kubernetes manifests.


## QoS classes

Kubernetes automatically assigns each Pod a **Quality of Service (QoS)
class** based on how requests and limits are configured. QoS classes
determine eviction priority when a node is under memory pressure.

| QoS class | Condition | Eviction priority |
|-----------|-----------|-------------------|
| **Guaranteed** | Every container has requests **and** limits set, and they are **equal** for both CPU and memory | Last to be evicted (highest priority) |
| **Burstable** | At least one container has requests or limits set, but they are **not** equal | Evicted after BestEffort |
| **BestEffort** | No container has any requests or limits | First to be evicted (lowest priority) |

### Guaranteed example

<TerminalBlock label="Guaranteed QoS">
{`resources:
  requests:
    cpu: "500m"
    memory: "256Mi"
  limits:
    cpu: "500m"
    memory: "256Mi"`}
</TerminalBlock>

Requests equal limits for both CPU and memory - this Pod is `Guaranteed`.

### Burstable example

<TerminalBlock label="Burstable QoS">
{`resources:
  requests:
    cpu: "250m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "256Mi"`}
</TerminalBlock>

Requests are lower than limits - this Pod is `Burstable`.

### BestEffort example

<TerminalBlock label="BestEffort QoS">
{`resources: {}`}
</TerminalBlock>

No requests, no limits - this Pod is `BestEffort` and will be the first
evicted under pressure.


## LimitRange and ResourceQuota

Cluster admins can enforce defaults and caps at the namespace level:

| Resource | Purpose |
|----------|---------|
| `LimitRange` | Sets default requests/limits for containers that don't specify them, and enforces min/max per container |
| `ResourceQuota` | Caps the **total** resources (CPU, memory, object count) that a namespace can consume |

These are namespace-scoped objects. They prevent teams from accidentally
requesting too many resources or forgetting to set requests entirely.


## Practical guidelines

- Always set **requests** - they are essential for proper scheduling
- Set **limits** for memory to prevent runaway processes from OOMKilling
  the node
- Be cautious with CPU limits - aggressive CPU throttling can cause latency
  spikes in latency-sensitive apps
- Use `Guaranteed` QoS for critical workloads that must not be evicted
- Use `LimitRange` to enforce sane defaults across a namespace
