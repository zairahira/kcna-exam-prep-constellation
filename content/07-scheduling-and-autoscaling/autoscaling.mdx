## Why autoscaling?

Static resource allocation wastes money when traffic is low and causes
outages when traffic spikes. Autoscaling adjusts capacity automatically
based on actual demand. Kubernetes supports autoscaling at multiple levels.


## Horizontal Pod Autoscaler (HPA)

The HPA scales the **number of Pod replicas** in a Deployment, ReplicaSet,
or StatefulSet based on observed metrics.

### How it works

1. The HPA controller queries the **Metrics API** at regular intervals
   (default: 15 seconds)
2. It compares the current metric value to the target you defined
3. It calculates the desired replica count and scales accordingly

### Creating an HPA

<TerminalBlock command="kubectl autoscale deployment web --cpu-percent=50 --min=2 --max=10">
{`horizontalpodautoscaler.autoscaling/web autoscaled`}
</TerminalBlock>

This creates an HPA that:
- Targets the `web` Deployment
- Keeps average CPU utilization at **50%**
- Scales between **2** and **10** replicas

### HPA as YAML

<TerminalBlock label="hpa.yaml">
{`apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50`}
</TerminalBlock>

### Key points

- The HPA requires **Metrics Server** to be installed in the cluster (it
  provides the `metrics.k8s.io` API)
- Pods must have CPU/memory **requests** set - utilization is calculated as
  `current usage / request`
- `autoscaling/v2` supports custom and external metrics (e.g. requests per
  second from Prometheus) in addition to CPU and memory
- Scale-up happens faster than scale-down (to avoid flapping)

<TerminalBlock command="kubectl get hpa">
{`NAME   REFERENCE        TARGETS   MINPODS   MAXPODS   REPLICAS   AGE
web    Deployment/web   35%/50%   2         10        3          5m`}
</TerminalBlock>


## Vertical Pod Autoscaler (VPA)

The VPA adjusts **resource requests and limits** for containers based on
historical and real-time usage.

### How it works

1. The VPA **recommender** analyzes resource usage over time
2. It recommends new request/limit values
3. Depending on the update mode, it either applies changes or just reports
   them

### Update modes

| Mode | Behavior |
|------|----------|
| `Off` | VPA only provides recommendations. No changes are applied. |
| `Initial` | VPA sets resources only when a Pod is created. Running Pods are not modified. |
| `Auto` | VPA evicts Pods and recreates them with updated resources. (This causes brief downtime for individual Pods.) |

### Key points

- VPA is **not** a built-in Kubernetes resource - it must be installed
  separately
- VPA and HPA should not target the same metric (e.g. both targeting CPU)
  because they will conflict
- VPA is useful for workloads where you don't know the right resource
  values upfront


## Cluster Autoscaler

The Cluster Autoscaler operates at the **node level**. It adds or removes
nodes from the cluster based on demand.

### Scale up

When a Pod cannot be scheduled because no node has enough resources, the
Cluster Autoscaler **provisions a new node** from the cloud provider (e.g.
adds an instance to an AWS Auto Scaling Group or a GKE node pool).

### Scale down

When a node is **underutilized** (its requested resources are below a
threshold, typically 50%) for a sustained period, the Cluster Autoscaler
drains and removes it.

### Key points

- The Cluster Autoscaler is **cloud-provider specific** - it integrates
  with AWS ASGs, GKE node pools, Azure VMSS, etc.
- It reacts to **unschedulable Pods** (scale up) and **underutilized
  nodes** (scale down)
- It does not monitor CPU or memory utilization directly - it looks at
  resource **requests** vs node capacity
- It works alongside HPA: HPA adds Pods → Pods become unschedulable →
  Cluster Autoscaler adds nodes

| Component | What it scales | Trigger |
|-----------|---------------|---------|
| HPA | Pod replicas | Metric thresholds (CPU, memory, custom) |
| VPA | Pod resources | Usage history and recommendations |
| Cluster Autoscaler | Nodes | Unschedulable Pods / underutilized nodes |


## KEDA - event-driven autoscaling

**KEDA** (Kubernetes Event-Driven Autoscaling) extends the HPA to scale
based on **external event sources**. It is a CNCF Graduated project.

### How it works

1. KEDA connects to an external system (a message queue, database, cloud
   service, etc.) via a **scaler**
2. It monitors the event source (e.g. queue depth, stream lag)
3. It activates or deactivates a Deployment by scaling replicas from 0 to N
   or back to 0

### Why KEDA?

The standard HPA can scale based on CPU, memory, or custom metrics exposed
through the Kubernetes Metrics API. KEDA goes further:

| Feature | HPA | KEDA |
|---------|-----|------|
| Scale to zero | No (minimum 1 replica) | Yes |
| Event sources | CPU, memory, custom metrics | 50+ built-in scalers (Kafka, RabbitMQ, AWS SQS, Prometheus, cron, etc.) |
| CNCF status | Built-in | Graduated project |

### Example use case

A Deployment processes messages from a Kafka topic. With KEDA:

- **No messages** → 0 replicas (saves resources)
- **Messages arrive** → KEDA scales up to handle the load
- **Queue drains** → KEDA scales back to 0

### Key KEDA concepts

| Concept | Purpose |
|---------|---------|
| `ScaledObject` | Links a Deployment to one or more event source triggers |
| `ScaledJob` | Links a Job to event source triggers (for batch processing) |
| `TriggerAuthentication` | Stores credentials for connecting to external systems |


## Summary

| Autoscaler | Level | Scales what | Trigger | Install |
|------------|-------|-------------|---------|---------|
| **HPA** | Pod | Replica count | CPU/memory/custom metrics | Built-in |
| **VPA** | Pod | Resource requests/limits | Usage history | Add-on |
| **Cluster Autoscaler** | Node | Node count | Unschedulable Pods | Cloud-provider add-on |
| **KEDA** | Pod | Replica count (including to 0) | External events | Add-on (CNCF Graduated) |
