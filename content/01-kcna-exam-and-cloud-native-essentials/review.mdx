## Module summary

This module introduced the KCNA certification and the foundational cloud native concepts you need before diving into Kubernetes itself. Here is what we covered and how it connects.

## The KCNA exam

The exam is 60 multiple-choice questions in 90 minutes, with a 75% passing score. The four domains are:

| Domain | Weight |
|---|---|
| Kubernetes Fundamentals | 44% |
| Container Orchestration | 28% |
| Cloud Native Application Delivery | 16% |
| Cloud Native Architecture | 12% |

Study proportionally — Kubernetes Fundamentals carries nearly half the exam weight.

## Cloud native principles

Cloud native is defined by five pillars: **containers**, **microservices**, **service meshes**, **immutable infrastructure**, and **declarative APIs**. These are not just buzzwords — they are the design principles that Kubernetes was built to support.

The key distinction is between **monolithic** and **microservice** architectures:

- Monoliths are simpler to build and debug but force all-or-nothing scaling and deployment
- Microservices allow independent scaling and deployment but introduce distributed system complexity

Kubernetes was designed for the microservice model — orchestrating many small, independent containers across a cluster.

## Containers and runtimes

Containers are isolated processes that share the host kernel. They are built from **layered images** stored in **registries**.

The **Container Runtime Interface (CRI)** is the standard API between Kubernetes and container runtimes. The two main CRI-compliant runtimes are:

- **containerd** — the most common, originally part of Docker
- **CRI-O** — built specifically for Kubernetes, the default in OpenShift

Docker was removed from Kubernetes in v1.24 because it does not implement CRI. This did not break existing workloads because containerd (which Docker uses internally) does implement CRI, and container images follow the **OCI standard** — they are not Docker-specific.

## Container orchestration

Container orchestration solves the problems that appear when you run containers at scale: scheduling, scaling, self-healing, networking, load balancing, and rolling updates.

**Docker Swarm** was Docker's built-in orchestrator. It was simple and familiar to Docker users but lacked extensibility, advanced scheduling, a pluggable architecture, and a broad ecosystem.

**Kubernetes** won because of:
- Battle-tested design from Google's Borg system
- Extensible architecture (CRI, CNI, CSI, CRDs)
- Vendor-neutral governance under the CNCF
- Universal cloud provider support (EKS, GKE, AKS)
- A massive community and ecosystem

## How these topics connect

The concepts in this module form a chain:

1. **Cloud native principles** define the *approach* — build loosely coupled, resilient, scalable systems
2. **Microservices** are the *architectural pattern* that implements these principles
3. **Containers** are the *packaging mechanism* that makes microservices practical
4. **Container runtimes** are the *engines* that run containers on a host
5. **Container orchestration** is the *system* that manages containers across a cluster
6. **Kubernetes** is the *orchestrator* that the industry standardized on

Everything that follows in this course builds on this foundation. The next module will take you inside Kubernetes itself — how its architecture works, what components make up a cluster, and how they communicate.
