## Why this matters

Containers share the host kernel. A Pod running as root with broad
capabilities is a path to compromising the node. Pod security policies
control what Pods are allowed to request at admission time.


## Pod Security Standards

Kubernetes defines three standardized policy **levels**:

| Level | Intended for | Allows |
|-------|--------------|--------|
| `privileged` | Trusted system workloads | Almost anything (privileged containers, host access) |
| `baseline` | Typical workloads | Blocks the worst escalations (no privileged containers, no hostPath) |
| `restricted` | Hardened production | Strict - requires non-root, drops capabilities, read-only FS, etc. |

These are **standards** defined in the Kubernetes docs, not Kubernetes
objects. They describe combinations of SecurityContext settings that
correspond to each level.


## Pod Security Admission

**Pod Security Admission (PSA)** is a built-in admission controller that
enforces the standards. You label a Namespace to declare its level and
the mode (enforce, audit, or warn):

<TerminalBlock command="kubectl label namespace analytics pod-security.kubernetes.io/enforce=baseline">
{"namespace/analytics labeled\n"}
</TerminalBlock>

Three modes per namespace:

- `enforce` - reject Pods that violate the level.
- `audit` - log violations but allow.
- `warn` - warn the user on creation but allow.

A typical pattern: `enforce: baseline` + `warn: restricted` - block the
worst, warn developers toward the strictest.

PSA replaced the older **PodSecurityPolicy (PSP)**, which was deprecated
in v1.21 and removed in v1.25.


## SecurityContext fields

A Pod (or individual container) can set a `securityContext` that PSA
evaluates. Common fields:

<TerminalBlock label="pod.yaml (snippet)">
{`spec:
  securityContext:
    runAsNonRoot: true          # fail if image's default user is UID 0
    runAsUser: 1000             # specific UID
    fsGroup: 2000               # group for mounted volumes
  containers:
  - name: app
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]
        add: ["NET_BIND_SERVICE"]`}
</TerminalBlock>

### Key fields

- **`runAsNonRoot: true`** - refuse to start if the container image&apos;s
  default user is root. Simple, high-leverage.
- **`readOnlyRootFilesystem: true`** - the container&apos;s root FS is
  read-only. Exploits that write to disk fail.
- **`allowPrivilegeEscalation: false`** - prevents `setuid` binaries and
  ambient capabilities from granting more privilege than the parent.
- **`capabilities.drop: ["ALL"]`** - drop all Linux capabilities. Add
  back only the specific ones the app needs (e.g. `NET_BIND_SERVICE` for
  binding low ports).


## The short list

For most production workloads:

1. Namespace labeled `enforce: baseline` (minimum) or `restricted` (better).
2. Pods set `runAsNonRoot: true`, `readOnlyRootFilesystem: true`,
   `allowPrivilegeEscalation: false`.
3. Drop all capabilities; add only what&apos;s needed.
4. No privileged containers unless unavoidable.
