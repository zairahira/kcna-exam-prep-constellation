## Workload identity

Humans authenticate to Kubernetes as **Users** (an external concept -
Kubernetes doesn&apos;t store users; identity providers do). Workloads
authenticate as **ServiceAccounts**.

A ServiceAccount is a namespaced Kubernetes object that represents a
Pod&apos;s identity when it calls the Kubernetes API.


## The default ServiceAccount

Every namespace has a `default` ServiceAccount. Pods that don&apos;t specify
one use it. The ServiceAccount&apos;s token is auto-mounted at
`/var/run/secrets/kubernetes.io/serviceaccount/token` inside the Pod.

You can disable this auto-mount for Pods that don&apos;t need API access:

<TerminalBlock label="pod.yaml (snippet)">
{`spec:
  automountServiceAccountToken: false`}
</TerminalBlock>

Or scope it to a specific ServiceAccount:

<TerminalBlock label="pod.yaml (snippet)">
{`spec:
  serviceAccountName: analytics-ingest
  # token for 'analytics-ingest' is mounted, not 'default'`}
</TerminalBlock>


## ServiceAccount + RBAC

A ServiceAccount by itself has no permissions. It becomes useful when you
bind a Role (or ClusterRole) to it.

<TerminalBlock label="rolebinding.yaml">
{`apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: analytics-ingest-pods
  namespace: analytics
subjects:
- kind: ServiceAccount
  name: analytics-ingest
  namespace: analytics
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io`}
</TerminalBlock>

Pods running as `analytics-ingest` can now perform whatever the
`pod-reader` Role allows.


## Token projection

Modern clusters use **bound, projected tokens** by default. The token
mounted into a Pod is audience-bound and expires automatically (typically
1 hour), with kubelet refreshing it. This is more secure than the long-lived
tokens used in older Kubernetes versions.


## Workload identity on the cloud

ServiceAccounts let workloads authenticate to the Kubernetes API - but
what about the **cloud provider&apos;s** APIs (S3, Cloud Storage, Secret
Manager)?

Rather than stuffing cloud credentials into Secrets, cloud providers offer
**workload identity** integrations that map a Kubernetes ServiceAccount to
a cloud IAM identity:

- **AWS**: IRSA (IAM Roles for ServiceAccounts) - SA → IAM role.
- **GCP**: Workload Identity - SA → Google Service Account.
- **Azure**: Azure AD Workload Identity - SA → Azure AD identity.

Setup varies, but the idea is the same: the Pod assumes the cloud role via
its ServiceAccount, no long-lived cloud credentials in the cluster.
