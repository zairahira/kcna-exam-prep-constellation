## Trusted image sources

The simplest attack against a cluster is tricking it into running a
malicious image. Defences:

- **Use a private registry** (internal Harbor, ECR, Artifact Registry)
  for your own images. Makes provenance clear.
- **Pin by digest**, not by tag: `image@sha256:abc...` is immutable. A
  tag like `latest` or `1.4` can be re-pushed by anyone with push access.
- **Pull from a narrow allowlist** of registries. Admission policy can
  block images from anywhere else.


## Image scanning

**Image scanning** tools inspect images for known vulnerabilities (CVEs)
in their layers - operating system packages, language libraries, etc.

Common tools:

- **Trivy** - open source, CLI or CI, scans images and filesystems.
- **Grype** - similar.
- **Snyk**, **Prisma Cloud**, **Aqua** - commercial platforms.

Scanning fits into two places:

- **CI pipeline** - fail the build if high-severity CVEs are found.
- **Registry** - continuous scans as new CVEs are published against
  already-pushed images.


## Image signing

Scanning tells you what&apos;s *inside* an image. Signing tells you
*who built it and whether it&apos;s been tampered with*.

**Cosign** (part of the **Sigstore** project, CNCF Incubating) is the
modern approach:

<TerminalBlock>
{`# Sign an image\ncosign sign ghcr.io/myorg/app:1.4.2\n\n# Verify\ncosign verify ghcr.io/myorg/app:1.4.2 --certificate-identity-regexp=...`}
</TerminalBlock>

Sigstore uses keyless signing backed by OIDC identities, avoiding
long-lived signing keys. Admission controllers can then require that
every image running in the cluster carry a valid signature from a trusted
identity.


## Admission controllers

When a request reaches the API server (and after authn/authz), it passes
through a chain of **admission controllers** before being persisted.
Admission controllers can reject a request or mutate it.

Two kinds:

- **Mutating admission** - modify the request (e.g. inject a sidecar,
  set a default label).
- **Validating admission** - accept or reject based on rules.

Many built-in admission controllers ship with Kubernetes (NamespaceLifecycle,
ResourceQuota, PodSecurity, etc.). You can also register **custom**
webhook-based admission controllers.


## Policy engines: OPA Gatekeeper and Kyverno

Two projects dominate custom admission policy:

- **OPA Gatekeeper** (CNCF Graduated via Open Policy Agent) - uses **Rego**,
  a declarative policy language. Very expressive. Steeper learning curve.
- **Kyverno** (CNCF Incubating) - Kubernetes-native policy as YAML. Easier
  to pick up; less expressive than Rego for complex cases.

Typical policies:

- "All images must come from `ghcr.io/myorg/*`."
- "All Pods must set `runAsNonRoot: true`."
- "Ingresses must have a TLS section."
- "No `hostPath` volumes in non-system namespaces."
- "All images must be signed by our CI OIDC identity."

Policy engines let you encode your security standards as code - and
enforce them at admission time, before a non-compliant object ever
lands in the cluster.
