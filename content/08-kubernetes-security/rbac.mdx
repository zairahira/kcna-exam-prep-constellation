## Why RBAC

Every request to the Kubernetes API is **authenticated** (who are you?)
and **authorized** (are you allowed to do this?). **Role-Based Access
Control (RBAC)** is the default authorization mechanism in Kubernetes.

RBAC answers: given a subject (user, group, or ServiceAccount), can it
perform `<verb>` on `<resource>` in `<namespace>`?


## The four objects

RBAC uses two pairs of objects - one for namespaced scope, one for
cluster scope.

| Object | Scope | Defines |
|--------|-------|---------|
| `Role` | Namespaced | Permissions within a single namespace |
| `ClusterRole` | Cluster-wide | Permissions across all namespaces or on cluster-scoped resources |
| `RoleBinding` | Namespaced | Grants a Role (or ClusterRole) to subjects, scoped to one namespace |
| `ClusterRoleBinding` | Cluster-wide | Grants a ClusterRole to subjects, cluster-wide |

A common pattern: define a `ClusterRole` once, and `RoleBinding` it into
many namespaces where you want its rules applied locally.


## Verbs

RBAC verbs are the API operations:

| Verb | Maps to |
|------|---------|
| `get` | Read a single object |
| `list` | List objects |
| `watch` | Subscribe to change events |
| `create` | Create a new object |
| `update` | Replace an object |
| `patch` | Partial update |
| `delete` | Remove an object |
| `deletecollection` | Delete many at once |

Special verbs exist for subresources (e.g. `pods/exec`, `pods/logs`) and
for escalation control (`bind`, `escalate` on roles).


## Example: read-only access to Pods in one namespace

<TerminalBlock label="role.yaml">
{`apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: analytics
rules:
- apiGroups: [""]      # "" = core API group
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]`}
</TerminalBlock>

<TerminalBlock label="rolebinding.yaml">
{`apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: zaira-pod-reader
  namespace: analytics
subjects:
- kind: User
  name: zaira@example.com
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io`}
</TerminalBlock>

Now the user `zaira@example.com` can read Pods and their logs in the
`analytics` namespace - and nothing else.


## Principle of least privilege

RBAC is additive. Subjects get the union of everything their bindings
grant them. Start from "no permissions" and grant only what&apos;s needed.

Anti-patterns to avoid:

- Granting `cluster-admin` ClusterRole to automation. Scope it down.
- Using wildcards (`resources: ["*"]`, `verbs: ["*"]`) unless truly
  necessary.
- Binding ServiceAccounts broadly across namespaces.

Check what a subject can do:

<TerminalBlock command="kubectl auth can-i create pods --as=zaira@example.com -n analytics">
{"yes\n"}
</TerminalBlock>
