## ReplicaSets

A **ReplicaSet** ensures that a specified number of identical Pod replicas
are running at any given time. If a Pod crashes or is deleted, the
ReplicaSet creates a replacement. If there are too many Pods, it terminates
the extras.

Key fields:

| Field | Purpose |
|-------|---------|
| `spec.replicas` | Desired number of Pod replicas |
| `spec.selector` | Label selector to identify which Pods the ReplicaSet manages |
| `spec.template` | The Pod template used to create new Pods |

You rarely create ReplicaSets directly. **Deployments** manage them for
you. The KCNA expects you to understand what a ReplicaSet does, but in
practice you work with Deployments.


## Deployments

A **Deployment** is the standard way to run stateless applications on
Kubernetes. It manages one or more ReplicaSets to provide:

- **Declarative updates** - change the Pod template and Kubernetes handles the rollout
- **Rolling updates** - new Pods are created before old ones are removed
- **Rollbacks** - revert to a previous version if something goes wrong
- **Scaling** - adjust the replica count up or down

### The hierarchy

```
Deployment
  └─ ReplicaSet (current)
       ├─ Pod-1
       ├─ Pod-2
       └─ Pod-3
```

The Deployment creates and manages ReplicaSets. Each ReplicaSet manages a
set of Pods. When you update the Pod template (e.g. change the image
version), the Deployment creates a **new** ReplicaSet and scales it up
while scaling the old one down.


## Defining a Deployment

<TerminalBlock label="deployment.yaml">
{`apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      containers:
        - name: web
          image: nginx:1.25
          ports:
            - containerPort: 80`}
</TerminalBlock>

<TerminalBlock prompt="$" command="kubectl apply -f deployment.yaml">
{"deployment.apps/web-app created\n"}
</TerminalBlock>

<TerminalBlock prompt="$" command="kubectl get deploy,rs,pods -l app=web-app">
{`NAME                      READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/web-app   3/3     3            3           30s

NAME                                 DESIRED   CURRENT   READY   AGE
replicaset.apps/web-app-6d9f4b5c7   3         3         3       30s

NAME                           READY   STATUS    RESTARTS   AGE
pod/web-app-6d9f4b5c7-abc12   1/1     Running   0          30s
pod/web-app-6d9f4b5c7-def34   1/1     Running   0          30s
pod/web-app-6d9f4b5c7-ghi56   1/1     Running   0          30s`}
</TerminalBlock>

Notice the naming chain: `web-app` (Deployment) → `web-app-6d9f4b5c7`
(ReplicaSet) → `web-app-6d9f4b5c7-abc12` (Pod).


## Rollout strategies

When you update a Deployment's Pod template, Kubernetes performs a
**rollout**. Two strategies are available:

### RollingUpdate (default)

New Pods are created incrementally while old Pods are terminated
gradually. At no point is the application fully offline. Two parameters
control the pace:

| Parameter | Default | Meaning |
|-----------|---------|---------|
| `maxUnavailable` | 25% | Maximum number of Pods that can be unavailable during the update |
| `maxSurge` | 25% | Maximum number of extra Pods that can exist above the desired count |

### Recreate

All existing Pods are killed **before** new ones are created. This causes
downtime but guarantees that two versions never run simultaneously. Use
this when your application cannot tolerate two versions running at the
same time (e.g. database schema conflicts).

<TerminalBlock label="Recreate strategy snippet">
{`spec:
  strategy:
    type: Recreate`}
</TerminalBlock>


## Managing rollouts

Trigger a rollout by updating the image:

<TerminalBlock prompt="$" command="kubectl set image deployment/web-app web=nginx:1.26">
{"deployment.apps/web-app image updated\n"}
</TerminalBlock>

Watch the rollout progress:

<TerminalBlock prompt="$" command="kubectl rollout status deployment/web-app">
{`Waiting for deployment "web-app" rollout to finish: 1 out of 3 new replicas have been updated...
Waiting for deployment "web-app" rollout to finish: 2 out of 3 new replicas have been updated...
deployment "web-app" successfully rolled out`}
</TerminalBlock>

View rollout history:

<TerminalBlock prompt="$" command="kubectl rollout history deployment/web-app">
{`deployment.apps/web-app
REVISION  CHANGE-CAUSE
1         <none>
2         <none>`}
</TerminalBlock>

Roll back to the previous version:

<TerminalBlock prompt="$" command="kubectl rollout undo deployment/web-app">
{"deployment.apps/web-app rolled back\n"}
</TerminalBlock>

Roll back to a specific revision:

<TerminalBlock prompt="$" command="kubectl rollout undo deployment/web-app --to-revision=1">
{"deployment.apps/web-app rolled back\n"}
</TerminalBlock>


## Scaling

<TerminalBlock prompt="$" command="kubectl scale deployment/web-app --replicas=5">
{"deployment.apps/web-app scaled\n"}
</TerminalBlock>

This updates `spec.replicas` on the Deployment, which passes the new
count to the ReplicaSet, which creates (or removes) Pods to match.
