## ConfigMaps

A **ConfigMap** stores non-sensitive configuration data as key-value pairs.
It lets you decouple configuration from container images, so you can change
settings without rebuilding.

### Creating a ConfigMap

Imperatively from literal values:

<TerminalBlock command="kubectl create configmap app-config --from-literal=LOG_LEVEL=info --from-literal=MAX_RETRIES=3">
{`configmap/app-config created`}
</TerminalBlock>

Imperatively from a file:

<TerminalBlock command="kubectl create configmap nginx-conf --from-file=nginx.conf">
{`configmap/nginx-conf created`}
</TerminalBlock>

Declaratively:

<TerminalBlock label="configmap.yaml">
{`apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: default
data:
  LOG_LEVEL: info
  MAX_RETRIES: "3"
  config.yaml: |
    database:
      host: db.example.com
      port: 5432`}
</TerminalBlock>

All values in a ConfigMap are strings. Note that `"3"` is quoted because
YAML would otherwise interpret it as an integer.


## Secrets

A **Secret** stores sensitive data: passwords, API keys, TLS certificates.
Structurally, Secrets are very similar to ConfigMaps, but Kubernetes treats
them slightly differently:

- Values are **base64-encoded** (not encrypted) by default
- Secrets can be **encrypted at rest** if you configure an encryption
  provider on the API server
- Access should be restricted via **RBAC** - anyone who can read Secrets in
  a namespace can see the decoded values

### Creating a Secret

<TerminalBlock command="kubectl create secret generic db-creds --from-literal=username=admin --from-literal=password=s3cretP@ss">
{`secret/db-creds created`}
</TerminalBlock>

Declaratively (values must be base64-encoded):

<TerminalBlock label="secret.yaml">
{`apiVersion: v1
kind: Secret
metadata:
  name: db-creds
type: Opaque
data:
  username: YWRtaW4=
  password: czNjcmV0UEBzcw==`}
</TerminalBlock>

You can use `stringData` to provide plain-text values instead - Kubernetes
will base64-encode them for you:

<TerminalBlock label="secret-stringdata.yaml">
{`apiVersion: v1
kind: Secret
metadata:
  name: db-creds
type: Opaque
stringData:
  username: admin
  password: s3cretP@ss`}
</TerminalBlock>

### Secret types

| Type | Purpose |
|------|---------|
| `Opaque` | Generic, default type. Arbitrary key-value pairs. |
| `kubernetes.io/tls` | TLS certificate and private key (`tls.crt`, `tls.key`). |
| `kubernetes.io/dockerconfigjson` | Registry credentials for pulling private images. |
| `kubernetes.io/service-account-token` | Automatically created for ServiceAccounts (legacy, pre-v1.24). |


## Consuming ConfigMaps and Secrets

Two main approaches: **environment variables** and **volume mounts**.

### As environment variables

<TerminalBlock label="pod-env.yaml">
{`apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
  - name: app
    image: myapp:1.0
    env:
    - name: LOG_LEVEL
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: LOG_LEVEL
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-creds
          key: password`}
</TerminalBlock>

You can also inject all keys at once with `envFrom`:

<TerminalBlock label="pod-envfrom.yaml">
{`envFrom:
- configMapRef:
    name: app-config
- secretRef:
    name: db-creds`}
</TerminalBlock>

### As volume mounts

Each key becomes a file in the mounted directory:

<TerminalBlock label="pod-volume.yaml">
{`apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
  - name: app
    image: myapp:1.0
    volumeMounts:
    - name: config-volume
      mountPath: /etc/config
      readOnly: true
  volumes:
  - name: config-volume
    configMap:
      name: app-config`}
</TerminalBlock>

This creates files like `/etc/config/LOG_LEVEL`, `/etc/config/MAX_RETRIES`,
and `/etc/config/config.yaml` inside the container.

Secrets work the same way - just replace `configMap` with `secret` in the
volume definition.


## Update behavior

This is a commonly tested difference:

| Method | Update behavior |
|--------|----------------|
| Environment variables | **Frozen at Pod start**. Changing the ConfigMap/Secret does NOT update env vars in running Pods. You must restart (delete/recreate) the Pod. |
| Volume mounts | **Eventually refreshed** by the kubelet. Changes propagate automatically, typically within 60-90 seconds (configurable via `--sync-frequency`). No Pod restart required. |

If your application watches config files on disk, volume mounts give you
live updates. If it reads env vars once at startup, you need to restart Pods
after a config change.


## base64 is not encryption

This is an important exam concept. Secrets are base64-encoded by default:

<TerminalBlock command="echo 'czNjcmV0UEBzcw==' | base64 --decode">
{`s3cretP@ss`}
</TerminalBlock>

Anyone who can `kubectl get secret db-creds -o yaml` can decode the values
instantly. base64 is an encoding, not a security measure.

To actually protect Secrets:

- **RBAC**: Restrict who can read Secrets in each namespace
- **Encryption at rest**: Configure `EncryptionConfiguration` on the API
  server so etcd stores Secrets encrypted
- **External secret managers**: Tools like HashiCorp Vault, AWS Secrets
  Manager, or the `external-secrets` operator keep secrets out of etcd
  entirely
