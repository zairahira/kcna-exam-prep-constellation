## Labels

A **label** is a key-value pair attached to a Kubernetes object's metadata.
Labels are the primary mechanism for **organizing, grouping, and selecting**
resources.

<TerminalBlock label="pod-with-labels.yaml">
{`apiVersion: v1
kind: Pod
metadata:
  name: web-frontend
  labels:
    app: storefront
    tier: frontend
    environment: production
spec:
  containers:
  - name: nginx
    image: nginx:1.25`}
</TerminalBlock>

Rules for labels:

- Keys can have an optional prefix (e.g. `app.kubernetes.io/name`). The
  prefix must be a DNS subdomain. The name part is max 63 characters.
- Values are strings, max 63 characters. Can be empty.
- Labels are **queryable** - this is what makes them powerful.

### Adding and removing labels imperatively

<TerminalBlock command="kubectl label pod web-frontend version=v2">
{`pod/web-frontend labeled`}
</TerminalBlock>

<TerminalBlock command="kubectl label pod web-frontend version-">
{`pod/web-frontend unlabeled`}
</TerminalBlock>

The trailing `-` removes a label.


## Selectors

A **selector** filters resources based on their labels. Two types exist:

### Equality-based selectors

Use `=`, `==`, or `!=` to match exact values:

<TerminalBlock command="kubectl get pods -l environment=production">
{`NAME            READY   STATUS    RESTARTS   AGE
web-frontend    1/1     Running   0          5h
api-server      1/1     Running   0          5h`}
</TerminalBlock>

<TerminalBlock command="kubectl get pods -l tier!=backend">
{`NAME            READY   STATUS    RESTARTS   AGE
web-frontend    1/1     Running   0          5h`}
</TerminalBlock>

### Set-based selectors

Use `in`, `notin`, and `exists` for more flexible matching:

<TerminalBlock command="kubectl get pods -l 'environment in (production, staging)'">
{`NAME            READY   STATUS    RESTARTS   AGE
web-frontend    1/1     Running   0          5h
api-staging     1/1     Running   0          3h`}
</TerminalBlock>

<TerminalBlock command="kubectl get pods -l 'tier notin (backend)'">
{`NAME            READY   STATUS    RESTARTS   AGE
web-frontend    1/1     Running   0          5h`}
</TerminalBlock>

Check whether a label key exists (regardless of value):

<TerminalBlock command="kubectl get pods -l 'environment'">
{`NAME            READY   STATUS    RESTARTS   AGE
web-frontend    1/1     Running   0          5h
api-server      1/1     Running   0          5h
api-staging     1/1     Running   0          3h`}
</TerminalBlock>

### Combining selectors

Comma-separated selectors are ANDed together:

<TerminalBlock command="kubectl get pods -l 'app=storefront,tier=frontend'">
{`NAME            READY   STATUS    RESTARTS   AGE
web-frontend    1/1     Running   0          5h`}
</TerminalBlock>

### Selectors in YAML

Different resources use different selector syntax:

<TerminalBlock label="Service selector (equality-based only)">
{`selector:
  app: storefront
  tier: frontend`}
</TerminalBlock>

<TerminalBlock label="Deployment selector (set-based with matchLabels / matchExpressions)">
{`selector:
  matchLabels:
    app: storefront
  matchExpressions:
  - key: environment
    operator: In
    values: [production, staging]`}
</TerminalBlock>

Key distinction: Services use simple `selector` maps (equality-based).
Deployments, ReplicaSets, and Jobs support both `matchLabels` and
`matchExpressions`.


## Annotations

An **annotation** is a key-value pair attached to a Kubernetes object, but
unlike labels, annotations are **not queryable**. They store arbitrary
metadata that tools, controllers, or humans can read.

<TerminalBlock label="pod-with-annotations.yaml">
{`apiVersion: v1
kind: Pod
metadata:
  name: web-frontend
  annotations:
    build.source/commit: "a1b2c3d"
    build.source/branch: "main"
    prometheus.io/scrape: "true"
    prometheus.io/port: "9090"
    description: "Main storefront frontend pod"
spec:
  containers:
  - name: nginx
    image: nginx:1.25`}
</TerminalBlock>

Annotations can hold:

- Larger values (no 63-character limit on values)
- Non-identifying metadata: build info, tool config, documentation
- Structured data that tools parse (e.g. Prometheus scrape config)


## Labels vs annotations

| | Labels | Annotations |
|---|--------|------------|
| Purpose | Identify and select objects | Store non-identifying metadata |
| Queryable | Yes (`-l` / selectors) | No |
| Value size | Max 63 characters | Larger (practical limit ~256 KB total) |
| Used by | Services, Deployments, scheduling, RBAC | Ingress controllers, monitoring tools, build pipelines |
| Example | `app: storefront` | `prometheus.io/scrape: "true"` |

Rule of thumb: if you need to **select or filter** by it, use a label. If
you need to **store metadata** that tools or humans read, use an annotation.


## Common labeling patterns

The Kubernetes documentation recommends a set of well-known labels under the
`app.kubernetes.io` prefix:

| Label | Purpose | Example value |
|-------|---------|---------------|
| `app.kubernetes.io/name` | Application name | `storefront` |
| `app.kubernetes.io/version` | Application version | `v2.1.0` |
| `app.kubernetes.io/component` | Component within the app | `frontend` |
| `app.kubernetes.io/part-of` | Higher-level application | `ecommerce-platform` |
| `app.kubernetes.io/managed-by` | Tool managing this resource | `helm` |

These are conventions, not requirements. Using them helps dashboards,
service meshes, and monitoring tools work consistently across teams.

### Showing labels in output

<TerminalBlock command="kubectl get pods --show-labels">
{`NAME            READY   STATUS    AGE   LABELS
web-frontend    1/1     Running   5h    app=storefront,environment=production,tier=frontend
api-server      1/1     Running   5h    app=storefront,environment=production,tier=backend`}
</TerminalBlock>

<TerminalBlock command="kubectl get pods -L app,tier">
{`NAME            READY   STATUS    AGE   APP          TIER
web-frontend    1/1     Running   5h    storefront   frontend
api-server      1/1     Running   5h    storefront   backend`}
</TerminalBlock>

The `-L` flag adds label values as extra columns - handy for quick scans.
