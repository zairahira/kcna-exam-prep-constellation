## The problem with static PVs

Pre-creating PVs manually (static provisioning) doesn&apos;t scale. Every
new workload needs an admin to hand-create a matching PV first.

**Dynamic provisioning** solves this: when a PVC is created, Kubernetes
automatically creates the PV (and the underlying storage) to satisfy it.


## What a StorageClass is

A **StorageClass** describes a "kind" of storage - its provisioner, its
parameters, and its reclaim policy. Example:

<TerminalBlock label="storageclass.yaml">
{`apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  encrypted: "true"
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer`}
</TerminalBlock>

A PVC references a StorageClass by name. When the PVC is created, the
named provisioner creates a PV on demand, sized and configured per the
StorageClass.

Clusters typically offer several classes: `fast-ssd`, `slow-hdd`,
`shared-nfs`, etc. Pick the class that matches the workload&apos;s needs.


## Default StorageClass

One StorageClass in the cluster can be marked as the **default** (via the
annotation `storageclass.kubernetes.io/is-default-class: "true"`).

PVCs without an explicit `storageClassName` use the default. This is
convenient but means "generic" PVCs inherit whatever the default is -
good for most cases, surprising when the default has unusual performance
or cost characteristics.

<TerminalBlock command="kubectl get storageclass">
{"NAME             PROVISIONER         DEFAULT\n"}
{"fast-ssd         ebs.csi.aws.com     false\n"}
{"standard (default) ebs.csi.aws.com   true\n"}
</TerminalBlock>


## `volumeBindingMode`

Two options control when the PV gets created:

- **Immediate** - provision the PV as soon as the PVC is created. Default.
- **WaitForFirstConsumer** - wait until a Pod is scheduled that uses the
  PVC, then provision the PV on a node that matches the Pod&apos;s
  constraints. Important for zonal storage (e.g. EBS volumes are
  zone-specific - if you provision before scheduling, you might provision
  in a zone that can&apos;t host the Pod).

`WaitForFirstConsumer` is usually the right choice.


## CSI - the Container Storage Interface

**CSI** is the standard API that Kubernetes uses to integrate with storage
systems. Any storage vendor that implements CSI can plug into Kubernetes
without modifying the core.

CSI replaced two older mechanisms (`in-tree` volume plugins and
`FlexVolumes`). All new storage integrations are CSI-based.

You usually don&apos;t interact with CSI directly. You install the CSI
driver your storage system provides, and the driver handles the rest.


## Summary of the flow

1. Admin installs a **CSI driver** (e.g. `ebs.csi.aws.com`).
2. Admin or distro creates one or more **StorageClasses** referencing the
   driver&apos;s provisioner.
3. Developer creates a **PVC** naming a StorageClass (or using the default).
4. Kubernetes calls the driver to create a **PV** and the underlying
   storage.
5. The PVC binds to the new PV. The Pod mounts the PVC.
