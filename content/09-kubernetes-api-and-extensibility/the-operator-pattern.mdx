## What is an operator?

An **operator** is a Kubernetes-native way to automate the management of
a complex application. It encodes **human operational knowledge** into
software that runs inside the cluster.

The formula is simple:

> **Operator = CRD + Custom Controller**

- The **CRD** defines a new resource type that represents the application
  or its configuration (e.g. `PrometheusRule`, `Certificate`, `PostgresCluster`).
- The **custom controller** watches instances of that CRD and takes
  action to make reality match the desired state.

The result: you manage complex stateful applications using the same
declarative `kubectl apply` workflow as everything else in Kubernetes.


## The reconcile loop

Every controller (including operator controllers) runs a **reconcile
loop**:

1. **Observe** - watch the API server for changes to the target resource.
2. **Diff** - compare the desired state (from the `spec`) with the
   actual state of the world.
3. **Act** - take whatever steps are needed to close the gap (create
   resources, update configs, restart processes, etc.).

This loop runs continuously. If something drifts - a Pod crashes, a
config changes, a new replica is requested - the controller detects it
and reconciles.

### How it works in practice

Suppose you have a database operator and you create this custom resource:

<TerminalBlock label="my-database.yaml">
{`apiVersion: db.example.com/v1
kind: PostgresCluster
metadata:
  name: orders-db
spec:
  version: "16"
  replicas: 3
  storage:
    size: 100Gi`}
</TerminalBlock>

The operator controller sees this CR and:

1. Creates a StatefulSet with 3 replicas running PostgreSQL 16.
2. Provisions PersistentVolumeClaims for 100Gi each.
3. Configures primary-replica streaming replication.
4. Sets up connection pooling and monitoring.
5. Continuously monitors health and performs failover if the primary goes down.

All of this from a simple YAML file. The operational knowledge is baked
into the controller.


## Real-world operators

| Operator | What it manages | CRD examples |
|----------|----------------|--------------|
| **Prometheus Operator** | Prometheus monitoring stack | `Prometheus`, `ServiceMonitor`, `PrometheusRule`, `Alertmanager` |
| **cert-manager** | TLS certificates (Let's Encrypt, Vault, etc.) | `Certificate`, `Issuer`, `ClusterIssuer` |
| **Strimzi** | Apache Kafka on Kubernetes | `Kafka`, `KafkaTopic`, `KafkaUser` |
| **CloudNativePG** | PostgreSQL clusters | `Cluster`, `Backup`, `ScheduledBackup` |
| **Rook** | Ceph storage | `CephCluster`, `CephBlockPool`, `CephObjectStore` |

These operators handle day-2 operations that would otherwise require
significant manual effort: upgrades, backups, scaling, failover, and
monitoring.


## Operator Framework and OLM

The **Operator Framework** (by Red Hat) provides tools for building and
managing operators:

| Component | Purpose |
|-----------|---------|
| **Operator SDK** | Scaffolding and development tools. Supports Go, Ansible, and Helm-based operators. |
| **Operator Lifecycle Manager (OLM)** | Manages the lifecycle of operators themselves - install, upgrade, and manage dependencies between operators. |
| **OperatorHub.io** | A public catalog of community and vendor operators. Browse and install operators for databases, monitoring, networking, and more. |

### Other operator-building tools

- **Kubebuilder** - a CNCF project for building controllers and operators
  in Go. The Operator SDK is built on top of Kubebuilder.
- **Metacontroller** - a lightweight approach where you write controller
  logic as webhooks (in any language).
- **KUDO** - Kubernetes Universal Declarative Operator, lets you build
  operators using declarative YAML plans.


## Operator maturity levels

The Operator Framework defines a **capability model** with five levels:

| Level | Capability | Example |
|-------|-----------|---------|
| 1 | Basic install | Automated deployment of the application |
| 2 | Seamless upgrades | Automated version upgrades with rollback |
| 3 | Full lifecycle | Backup, restore, failure recovery |
| 4 | Deep insights | Metrics, alerts, log processing |
| 5 | Auto-pilot | Auto-scaling, auto-tuning, anomaly detection |

Most production operators aim for levels 3-4. Level 5 is the aspirational
end state where the operator fully replaces a human operator.


## When to use operators

Operators make sense when:

- The application is **stateful** and has complex lifecycle needs
  (databases, message queues, monitoring stacks).
- Day-2 operations (backup, upgrade, failover) are **well-defined** but
  tedious to do manually.
- You want to offer a **self-service** experience - developers create a
  CR and get a fully managed instance.

For simple stateless apps, a standard Deployment is usually enough. You
do not need an operator for everything.
