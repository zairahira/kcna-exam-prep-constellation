## Everything is an API object

Every resource in Kubernetes - Pods, Deployments, Services, ConfigMaps,
Secrets, and more - is represented as an **API object**. When you run
`kubectl get pods`, you are querying the API server for Pod objects. When
you apply a YAML file, you are sending an API object to the server.

The API server is the **only** component that reads from and writes to
`etcd`. All other components (scheduler, controllers, kubelet) interact
with the cluster through the API server.


## The five top-level fields

Every Kubernetes API object shares a common structure:

| Field | Purpose |
|-------|---------|
| `apiVersion` | Which API group and version this object belongs to (e.g. `v1`, `apps/v1`) |
| `kind` | The type of object (e.g. `Pod`, `Deployment`, `Service`) |
| `metadata` | Identity info - `name`, `namespace`, `labels`, `annotations`, `uid` |
| `spec` | The **desired state** - what you want this object to be |
| `status` | The **actual state** - filled in by Kubernetes at runtime (you never write this) |

The `spec` is what **you** declare. The `status` is what **Kubernetes**
reports back. Controllers work to make `status` match `spec`.


## A complete manifest example

Here is a Deployment manifest that creates 3 replicas of an Nginx Pod:

<TerminalBlock label="deployment.yaml">
{`apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  namespace: default
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:1.27
          ports:
            - containerPort: 80`}
</TerminalBlock>

Breaking it down:

- **`apiVersion: apps/v1`** - Deployments live in the `apps` API group,
  version `v1` (stable).
- **`kind: Deployment`** - the type of object we are creating.
- **`metadata`** - the Deployment is named `nginx-deployment` in the
  `default` namespace and has a label `app: nginx`.
- **`spec.replicas`** - we want 3 copies of the Pod.
- **`spec.selector`** - the Deployment manages Pods whose labels match
  `app: nginx`.
- **`spec.template`** - the Pod template. This is itself a partial Pod
  spec (metadata + spec) that describes what each replica looks like.


## Applying a manifest

You send a manifest to the API server with `kubectl apply`:

<TerminalBlock command="kubectl apply -f deployment.yaml">
{`deployment.apps/nginx-deployment created`}
</TerminalBlock>

The API server validates the object, persists it to `etcd`, and
controllers take over from there (the Deployment controller creates a
ReplicaSet, which creates the Pods, which the scheduler assigns to nodes).


## Viewing the full object

Use `kubectl get` with `-o yaml` to see the complete API object,
including the `status` section that Kubernetes fills in:

<TerminalBlock command="kubectl get deployment nginx-deployment -o yaml" label="truncated output">
{`apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  namespace: default
  uid: a1b2c3d4-...
  ...
spec:
  replicas: 3
  ...
status:
  availableReplicas: 3
  readyReplicas: 3
  replicas: 3
  ...`}
</TerminalBlock>

Notice the `status` section - you did not write it. The Deployment
controller populated it after reconciling.


## Metadata fields to know

Inside `metadata`, you will commonly see:

| Field | Description |
|-------|-------------|
| `name` | Required. Unique within the namespace for this kind. |
| `namespace` | The namespace the object lives in. Defaults to `default`. Some objects (Nodes, PersistentVolumes) are cluster-scoped and have no namespace. |
| `labels` | Key-value pairs used for selection and grouping. |
| `annotations` | Key-value pairs for non-identifying metadata (e.g. build info, tool config). |
| `uid` | Auto-generated. Globally unique identifier for this object. |


## Imperative vs declarative creation

You can create objects two ways:

| Approach | Example | Use case |
|----------|---------|----------|
| **Imperative** | `kubectl create deployment nginx --image=nginx:1.27` | Quick one-off, experimentation |
| **Declarative** | `kubectl apply -f deployment.yaml` | Production, GitOps, reproducibility |

Declarative is the standard for production. The manifest is the
**source of truth** - it can be version-controlled, reviewed, and
reapplied safely.
