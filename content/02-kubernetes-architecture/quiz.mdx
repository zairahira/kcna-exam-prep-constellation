<Checkpoint
  id="m02-q4"
  question={<>The kube-scheduler has assigned a Pod to a node. Which component is responsible for actually starting the container?</>}
  choices={[
    { id: "a", label: "kube-scheduler" },
    { id: "b", label: "kube-controller-manager" },
    { id: "c", label: "kubelet" },
    { id: "d", label: "kube-proxy" },
  ]}
  correctId="c"
  feedback={<p>The scheduler only decides <em>where</em> a Pod runs by setting <code>spec.nodeName</code>. The <code>kubelet</code> on the chosen node watches for this assignment, pulls the container image via the container runtime (CRI), and starts the container. The scheduler never starts containers itself.</p>}
/>

<Checkpoint

/>

<Checkpoint
  id="m02-q6"
  question={<>What mechanism do Kubernetes components use to receive real-time updates from the API server without polling?</>}
  choices={[
    { id: "a", label: "Periodic HTTP GET requests on a timer" },
    { id: "b", label: "A watch - a long-lived HTTP connection that streams events as they happen" },

    { id: "d", label: "gRPC bidirectional streaming directly to etcd" },
  ]}
  correctId="b"
  feedback={<p>Components open a <strong>watch</strong> request to the API server - a long-lived HTTP connection. When a resource changes, the API server pushes an event to all watchers immediately. This event-driven design is what makes Kubernetes reactive. Components typically use <strong>informers</strong> that maintain a local cache updated via watches.</p>}
/>

<Checkpoint
  id="m02-q7"
  question={<>Why do <code>kube-scheduler</code> and <code>kube-controller-manager</code> use leader election in HA setups?</>}
  choices={[
    { id: "a", label: "To distribute work evenly across all replicas for better throughput" },
    { id: "b", label: "To ensure only one instance is actively processing at a time, preventing conflicting decisions" },
    { id: "c", label: "To allow each replica to manage a different subset of namespaces" },
    { id: "d", label: "To synchronise their in-memory caches with etcd" },
  ]}
  correctId="b"
  feedback={<p>Leader election ensures <strong>exactly one active instance</strong> at a time. If two schedulers tried to schedule the same Pod, you would get conflicts. The leader acquires a Lease object in the API server; standby replicas wait. If the leader fails, a standby takes over. This prevents split-brain while providing high availability.</p>}
/>

<Checkpoint

  question={<>The <code>kubelet</code> does not run as a container. Why?</>}
  choices={[
    { id: "a", label: "Container images for kubelet don't exist" },
    { id: "b", label: "It needs to manage the container runtime and node-level resources - it can't depend on the thing it manages" },
    { id: "c", label: "Kubernetes only supports Go binaries running as system processes" },
    { id: "d", label: "The API server rejects registration from containerised kubelets" },
  ]}
  correctId="b"
  feedback={<p>The <code>kubelet</code> is responsible for managing containers on the node - pulling images, starting/stopping containers, running health checks. If the kubelet itself ran as a container, it would depend on the container runtime it is supposed to manage. A failure in the runtime would take down the kubelet too, leaving the node unmanageable. It runs as a <code>systemd</code> service instead.</p>}
