<Checkpoint

  question="What does the kube-scheduler do when a new pod is created?"

<Checkpoint
  id="m02-q05"
  question="A Kubernetes cluster's API server is down for 30 minutes. All control plane and worker nodes are otherwise healthy. What happens to the pods already running on worker nodes?"
  choices={[
    { id: "a", label: "All pods are immediately terminated because the kubelet cannot talk to the control plane" },
    { id: "b", label: "Existing pods continue running, but no new pods can be scheduled and no status updates are recorded" },
    { id: "c", label: "Only pods on the control plane node keep running" },
    { id: "d", label: "The container runtime stops all containers until the API server comes back" },
  ]}
  correctId="b"
  feedback={<p>Kubernetes is designed to be resilient to control plane outages. The kubelet manages pods based on local state and keeps existing containers running even when it cannot reach the API server. kube-proxy's existing networking rules continue routing Service traffic. What breaks is anything requiring coordination: new scheduling, status updates, rolling updates, and new <code>kubectl</code> commands all fail until the API server is restored.</p>}
/>

<Checkpoint
  id="m02-q06"
  question="What does it mean that Kubernetes is a 'declarative' platform?"
  choices={[
    { id: "a", label: "You write imperative shell scripts that configure the cluster step-by-step" },
    { id: "b", label: "You describe the desired end state, and controllers continuously reconcile actual state toward it" },
    { id: "c", label: "All configuration must be declared in JSON format" },
    { id: "d", label: "Only administrators can declare new resources in the cluster" },
  ]}
  correctId="b"
  feedback={<p>Declarative means you describe <em>what</em> you want (the desired state, usually in YAML manifests), not <em>how</em> to get there. Kubernetes' controllers continuously compare desired state (from <code>spec</code>) to actual state (from <code>status</code>) and take action to close the gap. This reconciliation pattern is what makes Kubernetes self-healing.</p>}
/>

<Checkpoint

  choices={[
    { id: "a", label: "Each component polls etcd every few seconds" },
    { id: "b", label: "Each component opens a watch against the API server, which streams changes as they happen" },
    { id: "c", label: "Components broadcast UDP messages to each other when state changes" },
    { id: "d", label: "Components share a local database replicated by the kubelet" },
  ]}
  correctId="b"
  feedback={<p>Kubernetes components use the <strong>watch</strong> mechanism. A client opens a long-lived connection to the API server, receives the current state, and then gets a stream of change events as they happen. Nothing polls etcd directly (except the API server), and components do not talk to each other, they all use the API server's watch API.</p>}
/>
