## The orchestration problem

Running a single container on your laptop is easy. Running hundreds of
containers across dozens of machines in production is a completely different
challenge. You need answers to questions like:

- Which machine has enough CPU and memory to run this container?
- What happens if a machine goes down - who restarts the containers?
- How do containers find and talk to each other?
- How do you roll out a new version without downtime?
- How do you scale up when traffic spikes?

Solving these problems by hand (or with scripts) does not scale.
**Container orchestration** is the category of tools that automates all of
this. Kubernetes is the dominant orchestrator.


## Origins

Kubernetes traces back to **Borg**, Google's internal container orchestration
system that managed billions of containers per week. In 2014, Google
open-sourced a redesigned version as **Kubernetes** (Greek for "helmsman" or
"pilot"). The project was donated to the newly formed **Cloud Native Computing
Foundation (CNCF)** in 2015 and became its first Graduated project.

Key names to know:

| Term | What it is |
|------|-----------|
| `Kubernetes` | The open-source orchestrator (often abbreviated `K8s` - "K", 8 letters, "s") |
| `Borg` | Google's internal predecessor to Kubernetes |
| `CNCF` | The foundation that governs Kubernetes today |


## What Kubernetes does

Kubernetes provides a **declarative platform** for managing containerised
workloads and services. You tell it *what* you want (e.g. "run 3 replicas of
my web server"), and it figures out *how* to make that happen.

Core capabilities:

- **Scheduling** - places containers on nodes with sufficient resources
- **Self-healing** - restarts failed containers, replaces unresponsive nodes,
  kills containers that don't pass health checks
- **Horizontal scaling** - adds or removes container replicas based on load
- **Service discovery and load balancing** - gives each set of containers a
  stable DNS name and distributes traffic
- **Automated rollouts and rollbacks** - gradually updates your application
  and rolls back if something goes wrong
- **Storage orchestration** - automatically mounts storage systems (local,
  cloud, network) into containers
- **Secret and configuration management** - stores sensitive data like
  passwords and API keys separately from container images


## What Kubernetes does not do

This is tested on the KCNA. Kubernetes is a platform, not a complete solution:

| Kubernetes does NOT... | You need... |
|------------------------|-------------|
| Build container images | A build tool (Docker, Buildah, kaniko) |
| Provide CI/CD pipelines | Jenkins, GitHub Actions, Argo Workflows, Tekton |
| Deploy source code directly | A container image pushed to a registry first |
| Include built-in monitoring/logging | Prometheus, Grafana, Fluentd, OpenTelemetry |
| Manage your application code | Developers still write, test, and package apps |
| Provide a database or message queue | Run those as workloads or use managed services |

Kubernetes provides the **infrastructure layer** - it runs, scales, and heals
your containers. Everything above and below that layer is pluggable.


## The Kubernetes cluster

A Kubernetes **cluster** is the top-level unit. It consists of:

- A **control plane** - the brain that makes decisions (scheduling, detecting
  failures, responding to API requests)
- One or more **worker nodes** - the machines that actually run your
  containers

You interact with the cluster through the **Kubernetes API**, typically via
the `kubectl` command-line tool. The control plane exposes the API; worker
nodes receive instructions from it.
