  choices={[

If the scheduler were down, you wouldn&apos;t see a <code>FailedScheduling</code> event at all.</p>}

<DebugPuzzle
  id="m02-d3"
  syllabusIds={["KF-03"]}
  prompt={<p>After a control plane upgrade, you notice two instances of <code>kube-controller-manager</code> are running. The team worries both are processing work simultaneously. Is this a problem?</p>}
  artifact={
    <TerminalBlock command="kubectl get leases -n kube-system kube-controller-manager -o wide">
      {"NAME                      HOLDER                            AGE\nkube-controller-manager   ctrl-plane-2_abc12345-67890       2m\n"}
    </TerminalBlock>
  }
  choices={[
    { id: "a", label: "Yes - two controller managers will create duplicate Pods and corrupt the cluster." },
    { id: "b", label: "No - leader election ensures only one instance is active. The Lease object shows which replica is the leader; the other is on standby." },
    { id: "c", label: "No - controller managers are stateless, so running multiple instances simply doubles throughput." },
    { id: "d", label: "Yes - you must always run exactly one instance of every control plane component." },
  ]}
  correctId="b"
  explanation={<p>Both <code>kube-controller-manager</code> and <code>kube-scheduler</code> use <strong>leader election</strong> via Lease objects in the API server. Only the holder of the Lease actively processes work. The other replica is standby and takes over if the leader stops renewing. This is the standard high availability pattern. The <code>HOLDER</code> field confirms which instance is currently leading.</p>}
/>

    { id: "d", label: "kube-proxy on node-02 has been upgraded and is rebooting." },

  correctId="a"
  explanation={<p>The <code>kubelet</code> sends heartbeats (via <code>Lease</code> renewals and <code>NodeStatus</code> updates) to the API server. When heartbeats stop, the <strong>node controller</strong> (inside <code>kube-controller-manager</code>) marks the node <code>NotReady</code>. The endpoint controller then removes Pods on that node from Service endpoints, so <code>kube-proxy</code> stops routing traffic to them. The scheduler does not monitor node health - that is the node controller&apos;s job.</p>}
