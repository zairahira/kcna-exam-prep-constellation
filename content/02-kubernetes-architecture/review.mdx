## Module summary

This module moved from "containers running on machines" to "containers running under Kubernetes." You now know what a cluster is made of, which components do what, and how they coordinate.

## Cluster foundations

A **cluster** is one or more machines Kubernetes manages as a single unit. Machines are called **nodes** and come in two roles:

- **Control plane nodes** run the brain of the cluster
- **Worker nodes** run the application pods

A **pod** is the smallest deployable unit, one or more containers scheduled together, sharing a network namespace and storage. Pods, not containers, are what you deploy to Kubernetes.

## Control plane components

Five components make up the control plane:

| Component | Role | Key fact |
|---|---|---|
| **kube-apiserver** | Front door; every client and component talks to it | Stateless; can run multiple instances behind a load balancer |
| **etcd** | Cluster database | Only component that stores state; only the API server talks to it |
| **kube-scheduler** | Decides pod placement | Filters nodes, then scores them, does not start pods |
| **kube-controller-manager** | Runs reconciliation loops | Hosts many controllers in one process |
| **cloud-controller-manager** | Cloud provider integration | Optional, only present on cloud clusters |

The API server is the center of everything. The scheduler, controllers, kubelets, and kube-proxy all go through it. No component talks directly to another.

## Worker node components

Three components run on every worker node:

- **kubelet**: node agent; manages pods assigned to this node by watching the API server
- **kube-proxy**: implements Service networking on the node using iptables or IPVS
- **Container runtime**: pulls images and runs containers via CRI (containerd or CRI-O)

The kubelet is the reconciliation loop at the node level: "whatever pods the API server says belong on me, make them real and report their status."

## How the pieces communicate

Three ideas tie the architecture together:

**1. Hub-and-spoke.** The API server is the hub; every component is a spoke. Components are decoupled, they do not know about each other, only about the API server. This is what makes Kubernetes extensible: custom controllers (Module 10) plug in by watching the API server, no core changes needed.

**2. The watch mechanism.** Components do not poll. They open long-lived watches on the API server, which streams changes as they happen. This is what makes reconciliation feel instantaneous.

**3. Desired vs actual state.** Every object has a `spec` (what you want) and a `status` (what is observed). Controllers continuously compare them and take action to close the gap. This is the single most important mental model in Kubernetes.

## The flow when you deploy

When you `kubectl apply` a manifest, here is what happens:

1. `kubectl` sends the manifest to the **API server**
2. The API server authenticates you, validates the object, and writes it to **etcd**
3. A **controller** (e.g., the Deployment controller) sees the new object via watch and creates any dependent objects (a ReplicaSet, then pods)
4. The **scheduler** sees unscheduled pods and picks nodes for them
5. The **kubelet** on each chosen node sees the assignment, tells the **container runtime** (via CRI) to pull and run the containers
6. **kube-proxy** on every node updates networking rules so the new pods are reachable through any matching Services
7. Controllers keep watching. If anything drifts from desired state, pod crashes, node disappears, they trigger corrective action

Nothing in this flow requires you to coordinate steps. You declared the goal; the components pursued it.
