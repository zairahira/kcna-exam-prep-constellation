## The API server as single gateway

Every component in a Kubernetes cluster communicates through the
**kube-apiserver**. There is no direct communication between components.

- The scheduler does **not** talk to the kubelet directly
- Controllers do **not** read from `etcd` directly
- The kubelet does **not** tell the scheduler which Pods are running

Everything goes through the API server. This design gives Kubernetes a single
point to enforce **authentication**, **authorisation**, **admission control**,
and **audit logging**. It also means that if you secure the API server, you
secure the cluster's control path.

```
kubectl ──→ kube-apiserver ──→ etcd
                  ↑↓
     ┌────────────┼────────────┐
     ↓            ↓            ↓
scheduler   controllers    kubelet
```


## The watch mechanism

Kubernetes components do not constantly **poll** the API server asking "has
anything changed?" Instead, they use a **watch** - a long-lived HTTP connection
that streams changes as they happen.

How it works:

1. A component (e.g. the scheduler) opens a **watch** request to the API
   server for a specific resource type (e.g. Pods)
2. The API server holds the connection open
3. When a relevant object is created, updated, or deleted, the API server
   **pushes** an event to the watcher in real time
4. The component processes the event and takes action

This is why Kubernetes feels **reactive** - changes propagate quickly without
expensive polling loops.

### Informers and caches

In practice, components use a client library pattern called an **informer**:

- The informer opens a watch and maintains a **local in-memory cache** of the
  watched resources
- The component reads from the cache (fast) instead of hitting the API server
  for every read
- The watch keeps the cache up to date automatically
- If the watch connection drops, the informer **re-lists** (fetches the full
  state) and resumes watching

This pattern dramatically reduces load on the API server, which is critical
in large clusters with thousands of objects.


## Leader election

Some control plane components - specifically `kube-scheduler` and
`kube-controller-manager` - must run as **a single active instance** at any
time, even when multiple replicas exist for high availability.

If two schedulers both tried to schedule the same Pod simultaneously, you would
get conflicts and unpredictable behaviour. **Leader election** prevents this.

How it works:

1. All replicas of the component start up
2. They compete to acquire a **Lease object** in the API server
3. One replica wins - it becomes the **leader** and actively processes work
4. The other replicas are **standby** - they watch the Lease but do nothing
5. The leader periodically **renews** the Lease
6. If the leader crashes (stops renewing), a standby acquires the Lease and
   becomes the new leader

This provides **high availability** without **split-brain**: at most one
instance is active at any time.

<TerminalBlock command="kubectl get leases -n kube-system">
{`NAME                      HOLDER                                          AGE

kube-scheduler            ctrl-plane-1_def67890-...                       42d`}

The `HOLDER` column shows which replica is the current leader.


## End-to-end flow: creating a Pod

To bring it all together, here is what happens when you run
`kubectl create deployment nginx --image=nginx`:

| Step | Component | Action |
|------|-----------|--------|
| 1 | `kubectl` | Sends an HTTP POST to the API server to create a Deployment |
| 2 | `kube-apiserver` | Authenticates and authorises the request, runs admission controllers, stores the Deployment in `etcd` |
| 3 | **Deployment controller** | Watches Deployments. Sees the new one. Creates a ReplicaSet object via the API server |
| 4 | **ReplicaSet controller** | Watches ReplicaSets. Sees the new one. Creates a Pod object (with no `spec.nodeName`) via the API server |
| 5 | `kube-scheduler` | Watches for unscheduled Pods. Finds the new Pod. Evaluates nodes. Binds the Pod to a node by updating `spec.nodeName` via the API server |
| 6 | `kubelet` (on the chosen node) | Watches for Pods assigned to its node. Sees the new Pod. Pulls the `nginx` image via the container runtime (CRI). Starts the container. Reports status back to the API server |
| 7 | `kube-proxy` (on all nodes) | Watches for Service/Endpoint changes. If a Service selects this Pod, updates network rules to route traffic to it |

Every single step communicates through the API server. No shortcuts.


## Communication security

All communication between components and the API server is encrypted with
**mutual TLS (mTLS)**. Each component has its own certificate, and the API
server verifies client identities. This ensures:

- **Confidentiality** - no one can eavesdrop on cluster traffic
- **Integrity** - messages cannot be tampered with
- **Authentication** - each component proves its identity

The kubelet also uses TLS to secure its own API (which the API server calls
to fetch logs, exec into containers, etc.).
