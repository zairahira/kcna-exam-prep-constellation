## CI, CD, and CD

Three terms, often confused:

### Continuous Integration (CI)

Every code change is automatically built and tested. The goal: catch
breakage within minutes of the commit, not days later. CI enables many
small changes per day instead of risky big-bang merges.

### Continuous Delivery

Every change that passes CI is **automatically prepared for release** -
built, tested, packaged, and deployed to staging - but **production
deploy is a manual click**. The system is always in a deployable state.

### Continuous Deployment

Same as Continuous Delivery, but the **deploy to production is also
automated**. Every passing commit goes live. Requires strong test
coverage, feature flags, and monitoring.

Both Continuous Delivery and Continuous Deployment share the "CD"
acronym - context decides which is meant. Continuous Delivery is the
more common practice.


## Typical pipeline stages

A production pipeline typically looks like:

```
Commit → Build → Test → Security scan → Image build → Push → Deploy staging → Approval → Deploy prod
```

Breakdown:

| Stage | What it does |
|-------|-------------|
| **Build** | Compile, resolve dependencies |
| **Unit tests** | Fast, isolated tests |
| **Integration tests** | Tests involving real dependencies (DB, APIs) |
| **Security scan** | Image vulnerability scan (Trivy, Grype), SAST, dependency audit |
| **Image build** | Package into a container image |
| **Push to registry** | Upload the image, often tagged with the commit SHA |
| **Deploy to staging** | Apply manifests to staging cluster |
| **Smoke / E2E tests** | Verify the deploy works |
| **Production deploy** | Gated manually or automated |

Each stage is a gate: failure stops the pipeline and reports back.


## Why CI/CD matters in cloud native

Cloud native workflows depend on small, fast, reliable deploys. Without
CI/CD:

- Developers merge into long-lived branches → big-bang releases → rollbacks hurt.
- Manual deploys encode tribal knowledge (the "works on Dave&apos;s laptop" problem).
- Feedback loops from bug to fix are slow.

Kubernetes&apos;s rolling updates, immutable container images, and
declarative manifests make the deploy itself easy. The pipeline gets
code from commit to that point safely.


## Popular tools

### CNCF-hosted

- **Tekton** (CNCF Graduated) - pipelines as native Kubernetes CRDs.
  Pipeline = YAML; each Task runs in a Pod.
- **Argo Workflows** (CNCF Graduated) - general workflow engine on
  Kubernetes. Used for CI and data pipelines.
- **Jenkins X** - opinionated Kubernetes-native CI/CD built on other tools.

### External (not CNCF)

- **Jenkins** - classic, extremely popular, mature.
- **GitHub Actions** - integrated with GitHub repos. Managed runners.
- **GitLab CI** - integrated with GitLab repos.
- **CircleCI**, **Buildkite**, **Drone** - commercial/managed CI.

Many teams run their pipeline in a hosted service (GitHub Actions,
GitLab) and then hand off deploys to a GitOps tool (next lesson) running
inside the cluster.
