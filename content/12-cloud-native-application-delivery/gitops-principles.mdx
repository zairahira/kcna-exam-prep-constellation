## What is GitOps?

**GitOps** is a way of managing infrastructure and applications where
**Git is the single source of truth for desired state**. Changes go
through Git (pull request, review, merge); software running inside the
cluster continuously reconciles the cluster&apos;s actual state toward
what&apos;s in Git.

The term was coined at Weaveworks (~2017). It&apos;s now codified by the
**OpenGitOps** working group (part of the CNCF).


## The four principles

The OpenGitOps principles:

1. **Declarative** - the entire system is described declaratively
   (manifests, not scripts).
2. **Versioned and immutable** - desired state is stored in a way that
   enforces versioning and immutability (Git).
3. **Pulled automatically** - software agents automatically pull the
   desired state from the source.
4. **Continuously reconciled** - agents continuously observe actual
   state and converge it to desired.


## Pull vs push

### Push model (traditional CI/CD)

CI pipeline runs `kubectl apply` against the target cluster. The pipeline
needs credentials to write to every cluster.

<pre className="cc-card p-4 text-sm my-4 overflow-x-auto">{`Commit → CI pipeline → kubectl apply → Cluster`}</pre>

Drawbacks: cluster credentials live in CI; no continuous reconciliation
(config drift can go unnoticed); many clusters = many credentials.

### Pull model (GitOps)

A controller running **inside** the cluster watches a Git repo and applies
changes when the repo updates. The cluster reaches *out* to Git; Git (and
CI) never reaches *into* the cluster.

<pre className="cc-card p-4 text-sm my-4 overflow-x-auto">{`Commit → Git repo   ←── Controller in cluster (pulls, applies, reconciles)`}</pre>

Advantages: no inbound credentials needed, continuous reconciliation,
per-cluster scoping is natural.


## Why GitOps fits Kubernetes

Kubernetes is already declarative and continuously reconciling. GitOps
extends that model to the boundary: the cluster&apos;s *desired state
itself* lives in Git and is reconciled into the cluster.

- **Same mental model** - Git → controllers → reconcile loops, all the
  way down.
- **Declarative manifests** - K8s already speaks this.
- **Controller pattern** - GitOps tools are just another controller.


## Benefits

### Audit trail

Every change is a Git commit. You get `git log`, signed commits, review
history, issue links. Production state at any point maps to a Git SHA.

### Rollback

Rolling back = reverting the commit (or pointing the GitOps tool at an
earlier commit/tag). No special rollback tooling.

### Drift detection

If someone runs `kubectl edit` against production, the GitOps controller
sees a divergence between Git and cluster state and (depending on
config) alerts or automatically reverts.

### Disaster recovery

A destroyed cluster can be rebuilt from Git - create a new cluster,
install the GitOps controller, point it at the repo, wait. No
snapshots, no manual runbooks.


## Repo patterns

Common patterns:

- **Monorepo** - all apps, all clusters, one repo. Simple; can get
  crowded.
- **App-per-repo + config-repo** - application code in one repo, K8s
  manifests in another. Lets ops control deploys independently.
- **Cluster-per-repo / environment-per-branch** - clean separation by
  environment.

There&apos;s no single right answer - teams choose based on size,
review boundaries, and tooling.
