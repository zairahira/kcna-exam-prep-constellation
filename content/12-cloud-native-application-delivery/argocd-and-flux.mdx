## Two leading GitOps tools

Both **Argo CD** and **Flux** are CNCF **Graduated** projects. They solve
the same problem (GitOps for Kubernetes) with different styles.


## Argo CD

**Argo CD** is more **UI-driven** and **application-centric**. You
define `Application` CRDs that describe: "deploy this folder of
manifests from this repo to this cluster/namespace."

### Key concepts

- **Application** - a unit of deployment. Source (Git repo + path) +
  destination (cluster + namespace).
- **Sync** - the action of applying Git state to the cluster.
- **Health status** - derived from each resource&apos;s status (Deployments
  showing Progressing, Pods showing CrashLoopBackOff, etc.).
- **Sync status** - `Synced` (matches Git) or `OutOfSync` (drift).
- **Auto-sync** - automatically reconcile when Git changes. Optional.
- **App of Apps** - a pattern where one Application points at a folder of
  other Application manifests. Scales to many apps.

The web UI is Argo CD&apos;s defining feature: visual trees of what&apos;s
deployed, sync/health state, drift diffs, rollback buttons.

### Typical flow

1. Install Argo CD into the cluster.
2. Create an `Application` pointing at a Git path.
3. Argo CD clones the repo, diffs against the cluster, syncs if needed.
4. Developer pushes to Git → Argo CD detects and re-syncs.


## Flux

**Flux** is more **toolkit-style** and **composable**. Rather than one
`Application` CRD, Flux uses multiple CRDs for different concerns.

### Key concepts

- **GitRepository** - a reference to a Git repo + ref + interval.
- **HelmRepository** / **OCIRepository** - other sources.
- **Kustomization** - "apply the Kustomize build of this source at this
  path to the cluster." (Different from K8s&apos;s Kustomize CLI - same idea,
  a Flux resource.)
- **HelmRelease** - "install/upgrade this Helm chart with these values."

Flux doesn&apos;t ship with a primary web UI - it&apos;s CLI-first
(`flux` command). Teams often pair Flux with the Weave GitOps UI or
other dashboards.


## Common to both

- **Declarative** - you describe what should be deployed, not how.
- **Reconciliation loop** - watches the source, applies, re-syncs on
  change.
- **Multi-tenancy** - multiple teams/applications in one cluster.
- **Support for Helm and Kustomize** - both render manifests on the fly
  from these tools.
- **RBAC** scoping - who can deploy what where.


## Comparison

| | Argo CD | Flux |
|---|---------|------|
| Primary style | UI-driven, Application-centric | CLI-driven, toolkit-composable |
| Learning curve | Gentler - easier to see what&apos;s happening | Steeper - more CRDs, more to learn |
| Deployment model | Single controller, one-way pull | Multiple controllers (source, kustomize, helm, notification) |
| Multi-cluster | ApplicationSet + cluster registration | Multi-tenant patterns, often one Flux per cluster |
| Typical users | Teams that want visibility + self-service | Teams that want tight automation and composability |


## When to pick which

Both are excellent; the choice is often organisational.

**Argo CD** when:
- Developers should self-serve via a UI
- Visualising sync state for many apps matters
- You want the "App of Apps" pattern for hierarchical deployments

**Flux** when:
- You prefer CLI/CRD composition without a web UI dependency
- You want clean separation of source, kustomize, helm concerns
- You want multi-tenant isolation primarily via CRD boundaries

Teams also run both, or alternate as they learn.
