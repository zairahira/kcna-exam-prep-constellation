## Why Services exist

Pods are **ephemeral**. When a Pod is replaced (due to scaling, rolling
updates, or failures), the new Pod gets a different IP address. If your
frontend hard-codes a backend Pod's IP, it breaks the moment that Pod
restarts.

A **Service** provides a stable network identity for a set of Pods. It gives
you:

- A **stable virtual IP** (ClusterIP) that doesn't change
- A **stable DNS name** (e.g. `my-svc.default.svc.cluster.local`)
- **Load balancing** across all matching backend Pods


## How Services find Pods

A Service uses a **label selector** to identify which Pods it should route
traffic to. Kubernetes automatically maintains the list of matching Pod IPs
in an **Endpoints** (or **EndpointSlice**) object.

<TerminalBlock label="service.yaml">
{`apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  selector:
    app: my-app      # matches Pods with this label
  ports:
    - port: 80        # Service port
      targetPort: 8080 # Pod port
  type: ClusterIP`}
</TerminalBlock>

When a Pod with the label `app: my-app` is created, it is automatically added
to the `my-app` Endpoints list. When it's deleted, it's removed.

<TerminalBlock command="kubectl get endpoints my-app">
{`NAME     ENDPOINTS                           AGE
my-app   10.244.1.5:8080,10.244.2.3:8080     2m`}
</TerminalBlock>

**EndpointSlices** are the newer, more scalable version of Endpoints. They
break large endpoint lists into smaller chunks and are the default in modern
Kubernetes versions.


## Service types

### ClusterIP (default)

Creates an internal-only virtual IP. Only reachable from **within the
cluster**.

| Property | Value |
|----------|-------|
| Accessible from | Inside the cluster only |
| Gets a cluster-internal IP | Yes |
| Use case | Internal communication between services |

This is the most common type. If you don't specify a `type`, you get
`ClusterIP`.

### NodePort

Exposes the Service on a static port on **every node's IP**. Traffic to
`<NodeIP>:<NodePort>` is forwarded to the Service.

| Property | Value |
|----------|-------|
| Accessible from | Outside the cluster via any node's IP |
| Port range | 30000-32767 (by default) |
| Builds on | ClusterIP (a ClusterIP is also created) |
| Use case | Development, testing, simple external access |

<TerminalBlock label="nodeport-service.yaml">
{`apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  type: NodePort
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 30080   # optional - omit to auto-assign`}
</TerminalBlock>

### LoadBalancer

Extends `NodePort` by provisioning an **external load balancer** from the
cloud provider (AWS ELB, GCP LB, Azure LB). Gives you a public IP or DNS
name.

| Property | Value |
|----------|-------|
| Accessible from | The internet (via cloud LB) |
| Builds on | NodePort + ClusterIP |
| Use case | Production external access on cloud providers |
| Cost note | Each LoadBalancer Service creates a separate cloud LB (can be expensive) |

In production, teams often use a single Ingress or Gateway instead of many
LoadBalancer Services to avoid creating one load balancer per service.

### ExternalName

Maps a Service to an **external DNS name**. No proxying, no ClusterIP - just
a CNAME record.

<TerminalBlock label="externalname-service.yaml">
{`apiVersion: v1
kind: Service
metadata:
  name: my-database
spec:
  type: ExternalName
  externalName: db.example.com`}
</TerminalBlock>

When Pods resolve `my-database.default.svc.cluster.local`, they get a CNAME
to `db.example.com`. Useful for pointing to external databases or services
outside the cluster.


## Quick comparison

| Type | Internal | External | Cloud LB | Use case |
|------|----------|----------|----------|----------|
| `ClusterIP` | Yes | No | No | Default, internal comms |
| `NodePort` | Yes | Yes (via node IP) | No | Dev/test external access |
| `LoadBalancer` | Yes | Yes (via cloud LB) | Yes | Production external access |
| `ExternalName` | DNS alias | N/A | No | Pointing to external services |
