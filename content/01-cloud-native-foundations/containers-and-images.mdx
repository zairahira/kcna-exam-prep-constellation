## What is a container?

A **container** is a lightweight, isolated environment for running a process.
It bundles the application binary, its libraries, and its configuration into
a single unit that runs the same way everywhere - your laptop, a CI server,
or a Kubernetes node.

Containers use **OS-level isolation** (Linux namespaces and cgroups) rather
than hardware virtualisation. This makes them much lighter than VMs:

| | Container | Virtual Machine |
|---|-----------|----------------|
| Startup time | Milliseconds–seconds | Minutes |
| Size | MBs | GBs |
| Isolation | Process-level (shared kernel) | Hardware-level (own kernel) |
| Overhead | Minimal | Significant (hypervisor + guest OS) |
| Portability | Image runs anywhere with a container runtime | Needs compatible hypervisor |

Containers are **not** VMs. They share the host kernel. This is a key
distinction the KCNA tests.


## Container images

A **container image** is a read-only template used to create containers.
Think of it as a snapshot of a filesystem plus metadata (what command to
run, which port to expose, what user to run as).

### Layers

Images are built in **layers**. Each instruction in a `Dockerfile` adds a
layer:

<TerminalBlock label="Dockerfile">
{`FROM node:20-alpine       # base layer - Alpine Linux + Node.js
COPY package.json .        # layer 2 - your package.json
RUN npm install            # layer 3 - installed dependencies
COPY . .                   # layer 4 - your app code
CMD ["node", "server.js"]  # metadata - default command`}
</TerminalBlock>

Layers are **cached and shared**. If two images share the same base layer,
it's stored once on disk. Changing only the last layer (your app code)
doesn't re-download everything - only the changed layer is rebuilt.

### Tags

Images are identified by `<registry>/<repository>:<tag>`. For example:

- `docker.io/library/nginx:1.25` - NGINX version 1.25 from Docker Hub
- `ghcr.io/myorg/myapp:latest` - your app from GitHub Container Registry

The `latest` tag is **not special** - it's just a convention. It doesn't
auto-update. Pin to specific tags in production.


## Registries

A **container registry** is where images are stored and distributed.
Common registries:

- **Docker Hub** (`docker.io`) - the default public registry
- **GitHub Container Registry** (`ghcr.io`)
- **Amazon ECR**, **Google Artifact Registry**, **Azure Container Registry**
  - cloud-provider managed registries
- **Harbor** - open-source, self-hosted, CNCF Graduated project

Workflow: you **build** an image locally, **push** it to a registry, and
Kubernetes **pulls** it from the registry when it needs to run the container.


## The OCI specification

The **Open Container Initiative (OCI)** defines open standards for container
formats and runtimes. Two specs matter:

- **OCI Image Spec** - how images are structured (layers, manifests,
  config). Ensures images built by one tool can be run by any compliant
  runtime.
- **OCI Runtime Spec** - how containers are created and run. `runc` is the
  reference implementation.

The OCI exists so that the ecosystem isn't locked into a single vendor's
format. Docker, containerd, CRI-O, Podman - they all speak OCI.
