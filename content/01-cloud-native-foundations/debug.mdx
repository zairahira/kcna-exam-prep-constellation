<DebugPuzzle
  id="m01-d2"
  syllabusIds={["CNA-02"]}
  prompt={<p>A developer builds an image on macOS. It works locally but fails on the Kubernetes cluster with <code>exec format error</code>. What&apos;s the most likely cause?</p>}
  artifact={
    <TerminalBlock>
      {"exec /app/server: exec format error\n"}
    </TerminalBlock>
  }
  choices={[
    { id: "a", label: "The image was built for ARM (Apple Silicon) but the cluster nodes run x86_64 (AMD64). Architecture mismatch." },
    { id: "b", label: "The OCI spec doesn't support images built on macOS." },
    { id: "c", label: "The container registry rejected the push because the tag was 'latest'." },
    { id: "d", label: "containerd can't run images built by Docker." },
  ]}
  correctId="a"
  explanation={<p>Apple Silicon Macs build ARM64 images by default. If the cluster nodes are x86_64 (which most are), the binary inside the container is the wrong architecture. Fix: use <code>docker buildx build --platform linux/amd64</code> to cross-compile, or use multi-arch builds. Docker images are OCI-compatible and work on containerd - (d) is wrong.</p>}
/>

    { id: "c", label: "Kubernetes Pods are invisible to all command-line tools." },
    { id: "d", label: "The containers are running in a different namespace and docker can't see cross-namespace containers." },
  ]}
  correctId="a"
  explanation={<p>Kubernetes v1.24 removed <strong>dockershim</strong>. The runtime is now <code>containerd</code> or <code>CRI-O</code>, not Docker. <code>docker ps</code> talks to the Docker daemon, which isn&apos;t managing these containers. Use <code>crictl ps</code> (which speaks CRI) or <code>ctr</code> (containerd&apos;s CLI) to inspect containers on the node.</p>}
